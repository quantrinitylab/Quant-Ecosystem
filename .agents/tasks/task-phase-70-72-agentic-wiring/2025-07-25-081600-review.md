# Agentic session wiring and integration test suite for Phases 70-72

This change adds integration tests proving three platform gates: universal tool layer execution across 17 apps (Phase 70), voice-to-deploy and durable automation pipelines (Phase 71), and phone-free agentic living via a new `AgenticSession` class that wires voice-first-os to quant-tools (Phase 72). The single piece of production code is `AgenticSession`, a thin bridge composing `PhoneFreeController`, `IntentRouter`, and `ToolExecutor` into a voice-command-to-tool-execution flow. The tests are integration-level but execute against in-process mocks rather than real services.

**Watch for:** Hardcoded `permissions: 3` in AgenticSession bypasses the MCP permission-tier model (confirmed). The IntentRouter's keyword-matching approach means tests are tightly coupled to exact phrasing that triggers routing thresholds (confirmed). No error-path or timeout testing exists for the voice command execution flow (confirmed).

## High-level view

The Phase 70 test validates that the existing quant-tools infrastructure (registry, router, planner, executor, MCP adapter, audit) works end-to-end: 85+ tools register, NL intents route to correct apps, multi-step plans execute with dependency passing, the MCP adapter enforces auth and permission tiers, and every execution is audited.

The Phase 71 tests cover two independent pipelines. The codex test proves the full voice-to-scaffold-to-build-to-deploy lifecycle through `CodexVoiceInterface`. The automate test proves NL parsing to durable execution with checkpoint/resume semantics, validating that interrupted automations can be restarted at the correct step.

Phase 72 introduces the only new production code: `AgenticSession` in voice-first-os. It composes existing components to translate voice text into tool execution via intent routing, always running with maximum permission tier (3) and a fixed `'voice-user'` identity. The tests validate that five distinct apps can be invoked in a single voice-only session, but they don't test what happens when routing fails, when handlers throw, or when the session expires.

<details>
<summary>Issues (5)</summary>

1. **Hardcoded max-privilege execution** — `AgenticSession` uses `permissions: 3` for all voice commands regardless of tool tier requirements. Introduce a configurable permission level or delegate to the caller's auth context.
2. **Fixed identity for all voice users** — Every execution runs as `userId: 'voice-user'` with no caller-specific identity. Pass user identity into the constructor or `startSession()` to enable meaningful audit trails.
3. **No error-path coverage in Phase 72 tests** — No test verifies behavior when intent routing returns zero matches, when a handler throws, or when the session has expired. Add at least one negative-path test.
4. **Fragile routing keyword coupling** — Tests rely on specific word combinations (e.g., "deploy app studio infrastructure") to exceed the 0.1 confidence threshold. Any change to tool names, tags, or descriptions will silently break routing. Consider testing the router's match output explicitly rather than asserting downstream success.
5. **Session summary tracks apps but not failures** — `PhoneFreeController.getContextualResponse` increments `commandsExecuted` and records `appsUsed` on every call, including commands that fail at the executor level. The summary will report successful app usage for failed commands.

<summary>Details</summary>

## AgenticSession: privilege escalation by default

`AgenticSession.executeVoiceCommand` constructs a `ToolExecutionContext` with `permissions: 3` (the maximum tier) and `userId: 'voice-user'` hardcoded at lines 67-72 of `agentic-session.ts`. The MCP adapter (tested in Phase 70) correctly rejects calls with insufficient tiers, but the voice path bypasses that gate entirely.

```typescript
const context: ToolExecutionContext = {
  userId: 'voice-user',
  sessionId: this.sessionId,
  permissions: 3,
  dryRun: false,
};
```

A voice user saying "delete all files" would execute a tier-3 destructive operation with no additional confirmation. The constructor accepts `ToolDefinition[]` but no user identity or permission scope, so there's no path to parameterize this without changing the API.

## Routing fragility and the confidence threshold

The `IntentRouter` uses a keyword overlap scoring system with a 0.1 minimum confidence threshold. The FEAT-003 findings document acknowledges this explicitly: test commands had to be reworded to include routing keywords like "deploy app studio infrastructure" to match `quant-studio.deploy`. Test correctness depends on the exact wording of tool names, tags, and descriptions in the tool registry.

The Phase 72 tests assert only `r1.success === true` without verifying which tool was routed to. If a registry change causes "deploy app studio infrastructure" to route to a different tool that also has a registered handler, the test passes silently with the wrong behavior.

## Session state tracking asymmetry

`AgenticSession.executeVoiceCommand` calls `this.phoneFreeController.getContextualResponse(topMatch.appId, text)` after executor completion regardless of whether the execution succeeded or failed. The `getContextualResponse` method increments `commandsExecuted` and adds the app to `appsUsed` unconditionally. `getSessionSummary()` therefore reports usage metrics that don't distinguish successful from failed commands.

If a handler throws, the session summary would still show that app as "used" and count it as an executed command. The fix: move the `getContextualResponse` call inside the success branch.

## What's not tested

- No test for `executeVoiceCommand` when `router.route()` returns an empty array (the "no matching tool" error path exists in code but isn't exercised)
- No test for handler exceptions propagating through `AgenticSession`
- No test for session timeout behavior (`PhoneFreeController.isSessionExpired()` is never called by `AgenticSession`)
- No test for the `endSession()` -> re-use flow
- No test for concurrent voice commands (the executor is sequential per-call but nothing prevents parallel `executeVoiceCommand` calls)
- The Phase 70 plan-ordering test was weakened based on findings that `MultiStepPlanner` orders by relevance score, not logical dependency. The test verifies both drive and mail steps exist but not their relative order.
