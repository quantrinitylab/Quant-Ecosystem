# Wake word detection and privacy subsystem for Quant Live

This change adds four tightly-scoped modules to `@quant/quant-live`: a wake word detector with provider abstraction, an audio ring buffer with star/forget semantics, a privacy lamp controller that tracks active input sources, and an append-only privacy audit log. The implementation is deliberately minimal (443 LOC production) with the privacy lamp designed to be impossible to programmatically disable. All four modules follow the existing package conventions (callback-based subscription returning unsubscribe functions, types centralized in `types.ts`).

**Watch for:** `forget()` deletes starred entries, contradicting the star-as-preservation semantic (confirmed); module-scoped mutable ID counters shared across instances create cross-instance ordering issues (confirmed); `PrivacyAudit.clear()` undermines the append-only guarantee (confirmed); starred audio buffer entries accumulate without memory bound (confirmed).

## High-level view

The wake word detector uses a provider pattern: `WakeWordDetector` owns lifecycle and confidence filtering, delegating actual audio processing to a `WakeWordEngine` implementation. When no engine is set, the detector enters fallback mode without listening, meaning push-to-talk logic must live in a consumer that checks `isFallbackMode()`.

The privacy lamp controller computes state from a priority hierarchy (camera/screen > mic > wake-word > dormant) and deliberately exposes no method to suppress or disable the indicator. However, `unregisterSource()` can silently hide an active input if a caller unregisters while the underlying hardware is still active — the lamp trusts its callers.

The audio buffer is an unbounded array with periodic time-based filtering. Starred entries bypass purge and accumulate indefinitely, and `forget()` will delete starred entries (only `purge()` respects the starred flag). This asymmetry is likely a bug.

The privacy audit freezes events on write, achieving shallow immutability. The `clear()` method deletes the entire log, contradicting the "append-only" invariant. The API doesn't distinguish between user-initiated deletion and programmatic clearing.

<details>
<summary>Issues (6)</summary>

1. **forget() deletes starred entries** — `forget()` filters by timestamp only, ignoring `starred`. If a user stars a recent segment then calls `forget(10)`, the starred entries are destroyed. `purge()` respects `starred`; `forget()` should too. Add `&& !e.starred` to the filter or document that forget is destructive regardless of star status.
2. **Module-scoped ID counters** — `idCounter` in audio-buffer-log.ts and `auditIdCounter` in privacy-audit.ts are module-level globals. Multiple instances share the same counter, producing non-monotonic IDs per instance and order-dependent test behavior. Move counters to instance fields.
3. **clear() breaks append-only contract** — `PrivacyAudit.clear()` deletes the entire event log. If append-only is the security invariant, remove this method or gate it behind an explicit ceremony that itself gets logged as an audit event before wiping.
4. **Unbounded starred entry growth** — Starred `AudioBufferEntry` items with `Float32Array` payloads survive purge indefinitely. Add a `maxStarredEntries` or `maxTotalBytes` parameter to cap memory.
5. **PorcupineProvider is a duplicate energy detector** — The "real" Porcupine provider does the same energy threshold check as `EnergyBasedFallback`. The scaffold is fine but the `feedAudio` implementation is misleading since it can never actually detect a wake phrase — consider making it throw "not implemented" or documenting the integration path.
6. **No integration between modules** — Nothing records a `mic_activated` audit event when the lamp transitions to `active-listening`. The modules are individually correct but don't compose into the privacy narrative without a coordinator.

## forget() vs star asymmetry

The purge logic correctly preserves starred entries:

```typescript
private purge(): void {
  const cutoff = Date.now() - this.bufferDurationMs;
  this.entries = this.entries.filter((e) => e.starred || e.timestamp >= cutoff);
}
```

But `forget()` does not:

```typescript
forget(seconds: number): void {
  const cutoff = Date.now() - seconds * 1000;
  this.entries = this.entries.filter((e) => e.timestamp < cutoff);
}
```

A user stars a segment to preserve it, then invokes "forget last 10 seconds" expecting unstarred content to be deleted. Instead, everything in that window — including starred entries — is removed. The two methods have contradictory semantics around the same flag.

## Module-scoped mutable state

Both `audio-buffer-log.ts` and `privacy-audit.ts` use a module-level `let` counter for generating IDs:

```typescript
let idCounter = 0; // audio-buffer-log.ts
let auditIdCounter = 0; // privacy-audit.ts
```

Two `AudioBufferLog` instances in the same process share the counter. Instance A gets `buf-1`, instance B gets `buf-2`, instance A gets `buf-3`. Tests that create multiple instances depend on execution order for ID values. The fix is trivial: move the counter to an instance field.

## The "append-only" audit and its escape hatch

`PrivacyAudit.clear()` replaces the event array with `[]`, destroying the audit trail. For a privacy audit whose purpose is proving "what happened to user data," this is a significant design tension. Two paths: remove `clear()` entirely and treat the audit as genuinely append-only, or rename it to `userInitiatedPurge()` and record a `buffer_cleared` event before wiping.

## Audio buffer memory model

At 16kHz mono, 5 minutes of audio is ~9.6MB of Float32 data. Steady-state unstarred memory is bounded by `bufferDurationMs`. Starred entries have no expiration — a user who stars frequently during a long session accumulates entries indefinitely. The type doesn't include a byte-size field, so implementing a memory cap requires computing `data.length * 4` per entry.

## Privacy lamp trust boundary

`unregisterSource()` effectively silences the lamp for that source type. A caller could `unregisterSource('camera')` to hide the recording indicator while the camera is still physically active at a lower layer. The non-disable constraint prevents accidental suppression but doesn't prevent a malicious caller from unregistering. This is acceptable for an in-process trust model but worth documenting.

## Test coverage gaps

Missing coverage worth noting:

- `PorcupineProvider.init()` failure path when accessKey is missing (the throw)
- `EnergyBasedFallback` sustained-count detection logic (no test exercises the 3-frame sustained threshold)
- `forget()` interaction with starred entries (no test catches the bug described above)
- Concurrent rapid `append`/`purge` interleaving
