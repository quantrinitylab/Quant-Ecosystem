# QuantMeet video conferencing app scaffold

QuantMeet adds a complete Fastify+WebSocket backend for a video conferencing app with 7 services (room management, SFU media routing, recording, transcription, AI summary, action-item extraction, breakout rooms), 3 route files, and frontend type definitions. All state is in-memory Maps; the SFU is a structural simulation rather than a real media server. 102 tests exercise the service logic with genuine error-path coverage.

Watch for: WebSocket signaling has no authentication or room-existence validation (confirmed). The `recordings.ts` route casts the Fastify instance to extract a `storage` property that is never actually registered as a plugin (confirmed). The SFU service generates hardcoded ICE candidates pointing at 127.0.0.1 which is fine for a scaffold but will silently fail in any non-localhost deployment (confirmed).

## High-level view

The app scaffolding follows the monorepo pattern closely with one structural addition: `@fastify/websocket` registration for signaling. The `app.ts` listen callback explicitly types `err: Error | null`, diverging from the quantube pattern where inference handles it.

The room service is the most complete, with Zod validation on both create and join inputs, max-capacity enforcement, mute-on-entry policy, and proper lifecycle from active to closed. The SFU service simulates WebRTC transport negotiation with correct directional constraints and cascading cleanup.

The AI-dependent services (summary, action-items, transcript) depend on a locally-defined `AIInference` interface rather than importing a type from `@quant/ai`, so the contract with the real AI package is implicit rather than compile-time enforced.

WebSocket signaling handles the expected event vocabulary but performs no validation beyond JSON parsing. Any connected socket can broadcast to any room without proving room membership, and the WS-side room state is completely independent of `RoomService`.

The test suite tests real logic paths with error cases, boundary conditions, and cross-concern interactions. No route-level integration tests exist, and the WebSocket route has no test coverage.

## Unauthenticated WebSocket signaling

The `ws.ts` route accepts a raw WebSocket connection at `GET /ws` with no authentication middleware. On receiving a `join-room` message, it trusts the `participantId` field from the client payload without verification:

```typescript
case 'join-room': {
  if (!message.roomId || !message.participantId) {
    socket.send(JSON.stringify({ type: 'error', message: 'roomId and participantId required' }));
    return;
  }
  const connections = rooms.get(message.roomId) ?? [];
  connections.push({ socket, participantId: message.participantId, roomId: message.roomId });
```

There is no check that the room exists in `RoomService`, no check that the participantId corresponds to an authenticated user, and no check that the user has actually joined the room via the REST API. A malicious client can join any room to eavesdrop on signaling, send offers/answers/ICE candidates as any participantId, and broadcast to participants in rooms they shouldn't access.

The `rooms` Map inside `ws.ts` is completely independent of `RoomService`'s room state. A room closed via DELETE /rooms/:id will continue to accept WebSocket connections indefinitely.

## Storage plugin gap in recordings route

The `recordings.ts` route obtains a storage reference through an unsafe cast:

```typescript
const storage = (fastify as unknown as { storage: StorageClient }).storage;
const recordingService = new RecordingService(storage as StorageClient);
```

No Fastify plugin in `app.ts` or anywhere in the registration chain decorates the instance with a `.storage` property. At runtime, `storage` will be `undefined`. While `RecordingService.startRecording()` doesn't currently call storage methods, the `getStorage()` accessor would return undefined, and any future storage interaction will throw.

The monorepo pattern in `ci-runner` injects storage as a constructor parameter rather than pulling it from the Fastify instance.

## Recording lifecycle gap

`startRecording` sets status to `'recording'`, and `stopRecording` transitions to `'processing'`. The `Recording` interface declares four possible states (`'recording' | 'processing' | 'completed' | 'failed'`) but no code path ever reaches `'completed'` or `'failed'`. A client polling for recording status will never see a final state.

## Test coverage gaps

Not tested:

- No route-level tests (HTTP handler validation, response shapes, error propagation to HTTP status codes)
- No WebSocket integration tests (connection lifecycle, broadcast behavior, disconnect cleanup)
- `RecordingService` tests never exercise storage interaction — the mock is created but no test asserts any storage method is called, because the service doesn't actually call storage methods yet
