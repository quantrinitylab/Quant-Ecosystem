# Git hosting platform with AI DevTools for QuantMail

This change adds a complete GitHub-like git hosting backend: Prisma models (Repository, Branch, PullRequest, Review, Issue, BranchProtection, CIRun, CIJob), a standalone `git-server` service with Smart HTTP protocol, PR/Issue/Review/CI services in the quantmail backend, a CI runner worker service, six AI-powered DevTools services (code review, commit messages, PR descriptions, CI fix, code search, security scan), and Fastify routes wiring everything together. The approach splits the system into a low-level git protocol server (services/git-server) and higher-level domain services (apps/quantmail/backend), with the CI runner as a separate queue worker.

**Watch for:** Schema-route field mismatches that will cause runtime failures (confirmed: `isPrivate` vs `visibility`, `finishedAt` vs `completedAt`, missing `ciJobLog` model); git HTTP routes have no authentication gate (confirmed); issue/PR number generation has a race condition under concurrency (likely); AI DevTools routes accept unbounded diff payloads with no size limit (confirmed).

## High-level view

The route layer references fields and models that don't exist in the Prisma schema. Three confirmed mismatches will crash at runtime: the `git.ts` route writes `isPrivate` to Repository (which uses the `visibility` enum), the CI retry route resets `finishedAt` (the field is `completedAt`), and the CI log route queries `prisma.ciJobLog` (no such model exists).

The git-server service implements Smart HTTP protocol correctly ŌĆö proper pkt-line formatting, correct content types, stateless-rpc flags ŌĆö but the entire git transport layer has no authentication. Neither the info/refs endpoint nor the upload-pack/receive-pack POST handlers verify credentials, so any network client can clone and push to any repo.

The quantmail backend routes initialize PrismaClient as `null as unknown as PrismaClient`, making them compile-time stubs that will null-pointer at runtime. There's no guard or startup check that prevents the app from booting with these null references.

The AI DevTools services embed user-supplied content (diffs, logs, source code) directly into prompts with no size bounds. A large diff could exceed model context windows and produce silent truncation or failures.

The CI runner parses YAML configs and orders execution by stage, but the executor only simulates execution (logs `[OK]` for each script line without running anything). The worker connects to `localhost:6379` with no environment-based override.

Issue and PR number assignment uses `count + 1`, which races under concurrent writes. Both services fetch the current count and create with that number, risking duplicate-number collisions against the unique constraint.

## Schema-to-route field mismatches

Three routes reference fields or models that don't exist in the Prisma schema, all of which will produce runtime errors:

In `apps/quantmail/backend/routes/git.ts`, the `CreateRepoSchema` accepts `isPrivate: z.boolean().default(false)` and passes it directly to `prisma.repository.create({ data: { isPrivate: body.isPrivate, ... } })`. The Repository model has no `isPrivate` field ŌĆö it uses `visibility: RepoVisibility @default(PUBLIC)` with the enum `PUBLIC | PRIVATE | INTERNAL`. Prisma will reject the unknown field at runtime.

In `apps/quantmail/backend/routes/ci.ts`, the retry handler runs:

```typescript
const retried = await prisma.ciRun.update({
  where: { id: runId },
  data: { status: 'PENDING', startedAt: null, finishedAt: null },
});
```

The CIRun model defines `completedAt DateTime?`, not `finishedAt`.

In the same file, the job-logs endpoint queries `prisma.ciJobLog.findMany({ where: { jobId } })`. No `CIJobLog` model exists. The CIJob model has a `logs: String?` field suggesting logs are stored inline, not in a separate table.

## Unauthenticated git transport

The `git-http.ts` routes expose the full git protocol ŌĆö clone, fetch, and push ŌĆö with no authentication layer. A `GitAuthService` exists at `services/git-server/src/services/auth.ts` with proper token hashing and scope validation, but it's never wired into the HTTP routes:

- Anyone who can reach the server can clone any repo (including "PRIVATE" visibility repos)
- Anyone can push arbitrary commits to any repository

The receive-pack route is the critical gap. The auth service generates and validates personal access tokens with scopes like `repo:read` and `repo:write`, but no Fastify `preHandler` or middleware invokes it.

## Issue and PR number race condition

Both `IssueService.createIssue` and `PullRequestService.createPR` derive the next number via:

```typescript
const count = await this.prisma.issue.count({ where: { repoId } });
// then:
data: { number: count + 1, ... }
```

If two requests arrive concurrently for the same repo, both read the same count (e.g., 5), both attempt to create number 6, and the second insert hits the `@@unique([repoId, number])` constraint. The service has no retry logic. This needs either a database-level sequence, an atomic max-number query, or a retry-on-unique-violation wrapper.

## AI DevTools prompt size and injection

All six AI services embed user-supplied content directly into the prompt:

```typescript
prompt: `Review the following code diff...\n\nDiff:\n${validated.diff}`;
```

The `diff` field has no maximum length constraint. A 500KB diff would produce a prompt that exceeds most model context windows, causing silent truncation or excessive token costs. Adding `z.string().max(100_000)` to the input schemas would bound this. The prompt construction also doesn't mitigate prompt injection ŌĆö a crafted diff could include instructions that override the system prompt ŌĆö though output is validated with Zod which limits the blast radius.

## CI runner dead failure path and hardcoded connection

The `CIJobExecutor.executeJob` catch block can never trigger since the try block only pushes strings to arrays (never throws). The `failed` status path is dead code:

```typescript
for (const line of job.script) {
  stdoutLines.push(`$ ${expandedLine}`);
  stdoutLines.push(`[OK] ${expandedLine}`);
}
// catch block unreachable
```

The `main.ts` worker uses `connection: { host: 'localhost', port: 6379 }` hardcoded, ignoring environment configuration.

## Branch protection status check gap

`BranchProtectionService.enforceOnPush` checks `requiredApprovals` but never enforces `requireStatusChecks`:

```typescript
if (matchingRule.requiredApprovals > 0) {
  // checks approvals...
}
// requireStatusChecks is never checked
return { allowed: true };
```

A merge can proceed even when all CI checks have failed, as long as the approval count is met. The field is stored but never enforced ŌĆö this is a fail-open posture on the status-checks gate.

## getDiff returns a hardcoded stub

`PullRequestService.getDiff` returns a static string regardless of the actual PR:

```typescript
return `diff --git a/file.txt b/file.txt\n--- a/file.txt\n+++ b/file.txt\n@@ -1,3 +1,3 @@\n-old line\n+new line`;
```

Any consumer (including the AI code review service if wired together) will always see the same diff. No TODO or marker indicates this needs implementation.
