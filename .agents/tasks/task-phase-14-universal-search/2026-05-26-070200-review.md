# Universal Search: Hybrid BM25 + Vector with Cross-App Pipeline

This change adds a complete universal search system combining BM25 (MeiliSearch) and vector (Qdrant) search with a natural language query parser, Cohere-based reranking, post-query permission filtering, facet aggregation, and proactive contextual search. The implementation spans the existing `@quant/search` package (new services for each pipeline stage) and a new `search-indexer` microservice that consumes Kafka CDC events and dual-writes to both search backends. A SearchRouter class orchestrates the full pipeline end-to-end: parse query, embed, hybrid search, rerank, filter, facet, respond.

Watch for: The embedding provider in production `main.ts` is a zero-vector placeholder that will silently produce useless search results if deployed without replacement (confirmed). The permission filter operates post-query with no pre-query filtering, meaning Qdrant and MeiliSearch return unauthorized results that are only stripped afterward — at scale this can leak result counts and degrade performance (confirmed). The VectorClient makes unauthenticated HTTP calls to Qdrant with no API key or TLS enforcement (confirmed).

## High-level view

The search package gains seven new service modules that form a pipeline: VectorClient talks to Qdrant over plain HTTP, HybridSearchEngine fuses BM25 position-based scores with Qdrant cosine similarity scores using configurable weights (0.7/0.3 default), QueryParser uses regex to extract type/person/date from natural language, CohereReranker calls Cohere's rerank-v3 endpoint and falls back to original ordering on failure, PermissionFilter strips inaccessible results post-query, SearchFacetAggregator computes type/date/sender facets from filtered results, and ProactiveSearch runs cross-collection vector similarity for "related items" features.

The search-indexer service consumes CDC events from Kafka and dual-writes to MeiliSearch (text fields) and Qdrant (embeddings). Each content type has a dedicated handler with a Zod-validated schema. The `main.ts` runs a placeholder embedding provider that returns zero vectors — functional for tests but a deployment risk.

The SearchRouter in the API layer orchestrates the pipeline steps sequentially. Both routers hard-code `isAdmin: false` for all requests, meaning admin-level access is never granted regardless of the requesting user's actual role.

<summary>Issues (7)</summary>

1. **Zero-vector embedding placeholder in main.ts** — The production entry point returns `new Array(1024).fill(0)` for all embeddings. Any deployment without replacing this provider will produce meaningless vector search results with no visible error. Wire the RoutingTable integration or fail loudly at startup if no real provider is configured.
2. **Qdrant accessed without authentication or TLS** — VectorClient constructs a plain `http://host:port` URL with no API key header. Any network-adjacent attacker can read, modify, or delete all indexed vectors. Add API key support and default to HTTPS in non-dev environments.
3. **Post-query-only permission filtering** — Both MeiliSearch and Qdrant return the full top-50 results including private documents; permissions are only enforced after retrieval. This leaks total counts, can return fewer than `limit` results after filtering (no backfill), and does O(n) filtering on every query. Push tenant/visibility filters into the Qdrant `filter` clause and MeiliSearch `filter` parameter for defense in depth.
4. **Hard-coded `isAdmin: false` in routers** — Both SearchRouter and ProactiveRouter pass `isAdmin: false` to the permission filter regardless of the authenticated user's role. Admin users will have their results incorrectly filtered. Accept the admin flag from the auth context.
5. **No backfill after permission filtering** — The pipeline fetches 50 results, reranks to `limit` (default 10), then filters by permissions. If 5 of 10 are private, the response contains only 5 results with no mechanism to fetch replacements. Consider over-fetching or iterating until the requested limit is met.
6. **Kafka message processing swallows errors silently** — The `eachMessage` handler catches all exceptions and logs them but does not dead-letter or retry. Transient failures (Qdrant timeout, MeiliSearch overload) cause permanent data loss in the search index. Add a retry/DLQ strategy.
7. **BM25 score normalization uses positional rank, not MeiliSearch scores** — HybridSearchEngine computes BM25 scores as `1 - idx/length` (rank-based), discarding MeiliSearch's actual relevance scores. This means the 2nd result always scores ~0.98 regardless of how close it is to the 1st. If MeiliSearch returns scored hits, use those scores for more meaningful fusion.

## Qdrant integration without auth or transport security

VectorClient constructs its base URL from a host and port with no TLS option and no API key header. Every `fetch` call goes over plain HTTP. In production, any service on the same network can issue arbitrary queries, upsert poisoned vectors, or delete collections. Qdrant supports both API key authentication and HTTPS — neither is wired in.

```typescript
constructor(host: string, port: number = 6333) {
  this.baseUrl = `${host}:${port}`;
}
```

There is no timeout configuration or retry logic on the HTTP calls. A slow Qdrant response will block the calling handler indefinitely since `fetch` has no default timeout in Node.

## Placeholder embedding provider deployed as production code

The `main()` function in search-indexer creates an `EmbeddingProvider` that returns 1024-dimensional zero vectors for every input:

```typescript
const embeddingProvider: EmbeddingProvider = {
  embed: async (texts: string[]) => {
    return texts.map(() => new Array(1024).fill(0) as number[]);
  },
};
```

This runs unconditionally at startup. Every document indexed by this service will have an identical zero vector, making vector search return arbitrary results (all cosine similarities are undefined for zero vectors). There's no environment variable gate, no startup error, no fail-fast check. The comment says "real implementation uses RoutingTable" but nothing enforces that transition.

## Hybrid search score fusion asymmetry

HybridSearchEngine normalizes BM25 scores using positional rank (`1 - idx / length`) rather than MeiliSearch's actual relevance scores. The code treats hits as an array of equally-spaced ranks. Vector scores from Qdrant, by contrast, are actual cosine similarity values normalized via min-max. The fusion therefore mixes a linear rank distribution (BM25 side) with an actual similarity distribution (vector side): a BM25 hit at position 2 out of 50 always gets score ~0.96, even if MeiliSearch considers it barely relevant.

## Permission filter: post-query only, with no backfill

The pipeline fetches 50, reranks to `limit` (typically 10), then applies permission filtering. If 4 of those 10 are private, the user gets 6 results. There is no loop to fetch more, no over-fetching heuristic, and no pre-query filter pushed into Qdrant or MeiliSearch.

The Qdrant filter clause in HybridSearchEngine's options supports `must` and `must_not` conditions, so pushing a visibility filter pre-query is already possible with the existing infrastructure — it's just not wired.

Both SearchRouter and ProactiveRouter hard-code:
```typescript
{ userId: validated.userId, isAdmin: false }
```
The admin path in PermissionFilter is dead code. No request will ever trigger admin bypass.

## Query parser: multi-word name truncation

The person-reference regex `\bfrom\s+(\w+(?:\s+\w+)?)\b` captures at most two words after "from". "From John Smith" works but "from John van der Berg" captures only "John van". Queries involving multi-word names will produce incorrect filter values.

The `last N days` pattern uses a type assertion to pass the regex match array to `resolve`:
```typescript
dateRange = (pattern.resolve as (m?: RegExpMatchArray) => DateRange)(match);
```
Only one pattern actually uses the match argument, but the cast obscures which patterns are parameterized and which are not.

## Kafka consumer: no retry, no DLQ, no backpressure

The `eachMessage` handler catches all exceptions and logs them. Failed events are permanently lost from the search index — no dead-letter queue, no retry mechanism, no circuit breaker. If Qdrant is temporarily unavailable, every event during that window creates a consistency gap between MeiliSearch and Qdrant that is never reconciled.

No concurrency control exists. If events arrive faster than downstream services can absorb them, handlers pile up without backpressure.

## Dual-write consistency gap in handlers

Each handler writes to MeiliSearch first, then Qdrant. If the MeiliSearch write succeeds but the Qdrant write fails (or vice versa), the document becomes searchable in one system but not the other. There's no compensation mechanism and no reconciliation process.

The `if (vector.length > 0)` guard means empty-text documents are indexed in MeiliSearch but skipped for Qdrant. BM25 can find these documents but vector search cannot, creating a permanent asymmetry in hybrid results for those items — they'll always receive a vectorScore of 0 in the fusion.

## Test coverage gaps

- No integration test verifying the full pipeline end-to-end with realistic data shapes
- PermissionFilter's `groupIds` field exists in the schema but is never checked in `isAccessible` — group-based access is silently ignored
- QueryParser has no tests for adversarial input (extremely long queries, unicode edge cases)
- No test for concurrent handler execution or partial-failure scenarios (MeiliSearch succeeds, Qdrant fails)
