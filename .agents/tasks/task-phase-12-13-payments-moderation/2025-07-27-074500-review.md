# Fraud detection, ad billing, agent spending, and moderation services

Phase 12 adds four payment-domain services (fraud detection, ad billing, agent spending limits, dispute management) and Phase 13 adds seven moderation-domain services (abuse graph, spam detection, AI output safety, ad policy enforcement, bot detection, content labels, safety audit log). All eleven services follow the same pattern: class with in-memory Maps, Zod validation at the boundary, and a co-located test suite.

**Watch for:** Fraud detection's `checkTransaction` is disconnected from `recordTransaction` — a checked transaction isn't automatically recorded, meaning signals are based on stale state if callers forget to call both (confirmed). The agent spending limit service has a TOCTOU gap between `checkSpend` and `recordSpend` — a concurrent caller can pass the check while another is between check and record (confirmed). The `transactionHistory` array in FraudDetectionService grows without bound (confirmed).

## High-level view

The payments additions center on four independent services that don't reference each other or the existing ledger/wallet services. FraudDetectionService accumulates per-user transaction history, device fingerprints, and country data in memory, then scores incoming transactions against velocity, amount, device, and geo signals. The two-call API (`checkTransaction` then `recordTransaction`) means a forgotten `recordTransaction` call leaves the fraud model permanently stale for that user.

AdBillingService manages campaign lifecycle with CPM/CPC/CPA billing and enforces both daily and total budget caps. When a budget is exhausted the campaign transitions to an `exhausted` state and silently returns `null` instead of billing — an asymmetry with "campaign not found" which throws.

AgentSpendingLimitService implements per-agent budgets with a check-then-spend pattern. The gap between `checkSpend` (read-only) and `recordSpend` (mutating) is a classic TOCTOU pattern — safe in single-threaded tests but fragile if ever called from concurrent request handlers.

The moderation services are lighter in state. AbuseGraphService builds an adjacency list and runs BFS for connected-component cluster detection. SpamDetectionService uses Jaccard similarity on word sets for fuzzy duplicate detection — functional for copy-paste bots but trivially defeated by word insertion. AIOutputSafetyService runs regex-based PII patterns (SSN, credit card, email only — phone numbers and other PII pass through). SafetyAuditLogService uses `Object.freeze` for append-only immutability but copies the full entry array on every query call.

<details>
<summary>Issues (5)</summary>

1. **Check-without-record in FraudDetectionService** — `checkTransaction` does not call `recordTransaction` internally, so a caller who checks but forgets to record leaves the fraud model permanently behind. Either auto-record on check, or document the two-step contract prominently.
2. **Unbounded history growth** — `FraudDetectionService.transactionHistory` is a plain array that never evicts old entries. In any long-lived process this becomes a memory leak. Add a sliding window or max-size eviction.
3. **TOCTOU in AgentSpendingLimitService** — `checkSpend` then `recordSpend` as separate calls allows concurrent spends to both pass the limit check before either records. A single atomic `attemptSpend` method would close the gap.
4. **Ad campaign daily reset reactivation churn** — `resetDailySpend` reactivates campaigns where `spent < budget`, but does not check whether the _remaining_ budget is enough to serve even one event at the campaign's configured rates. A campaign with $0.001 remaining gets reactivated, serves one impression, and immediately exhausts again.
5. **AIOutputSafetyService PII regex is minimal** — Only SSN, credit card, and email patterns are checked. Phone numbers, passport numbers, and other PII types pass through undetected. Document as a known gap if intentional.

## Fraud detection signal architecture

FraudDetectionService scores transactions against four signals — velocity (40 points), amount anomaly (35 points), device anomaly (25 points), and geo anomaly (30 points) — summed and capped at 100. The score-to-action mapping has a hardcoded `review` action at score >= 30 that sits below the configurable `flag` threshold:

```typescript
if (score >= this.config.riskThresholds.block) return 'block';
if (score >= this.config.riskThresholds.flag) return 'flag';
if (score >= 30) return 'review';
return 'allow';
```

If a caller configures `flag` below 30, the `review` action becomes unreachable. The hardcoded 30 acts as a floor but creates a non-obvious interaction with the configurable thresholds.

The `transactionHistory` array grows forever. Every `recordTransaction` pushes to an unbounded array that `checkVelocity` and `checkAmountAnomaly` scan with `.filter()` on every check. In a long-running process this degrades both memory and CPU linearly with transaction volume.

## Agent spending: check-then-spend gap

The `checkSpend` / `recordSpend` separation lets callers ask "can I?" before committing external side effects. But two concurrent handlers can both call `checkSpend`, both get `allowed: true`, and both call `recordSpend`, pushing the agent over its limit. The `hourlySpent` / `dailySpent` / `monthlySpent` counters are mutated directly on the budget object with no locking or atomic compare-and-set.

A single `attemptSpend` method that atomically checks and records (returning success/failure) would close this gap without changing the external contract for callers who don't need the two-step pattern.

## Ad billing budget edge cases

`resetDailySpend` reactivates exhausted campaigns where `spent < budget`. But an exhausted campaign might have been exhausted by hitting the daily cap, not the total cap. After reset, the daily counter is zero but the total might be just barely under budget. The campaign gets reactivated, serves one impression, and immediately exhausts again — a pointless cycle that generates a billing record each time.

The asymmetry between "campaign not found" (throws) and "campaign exhausted during this call" (returns null) could surprise integrators who don't check for null returns from `recordImpression`/`recordClick`/`recordConversion`.

## Spam detection: check-before-record ordering vulnerability

The `recordMessage` and `checkContent` are separate calls. A spammer's messages are only detectable as duplicates after `recordMessage` has been called for prior messages. If a burst of identical messages arrives and are all checked before any are recorded, none will be flagged as duplicates. This is the same pattern as the fraud service — the check doesn't see state that hasn't been explicitly recorded yet.

## Safety audit log query cost

The `query` method copies the full array (`[...this.entries]`) on every call before filtering. For a log that only grows, this becomes expensive over time. The `limit` filter applies after all other filters using `.slice(-limit)` (takes last N matching entries), which is undocumented "most recent N" semantics rather than "first N results".

## Test coverage gaps

Not tested: concurrent access patterns (the TOCTOU gap in agent spending), behavior after very large history accumulation (memory/perf), campaign reactivation edge cases after `resetDailySpend`, the interaction between `checkContent` and `recordMessage` ordering in spam detection, and `Object.freeze` enforcement in the audit log (that mutations actually throw).
