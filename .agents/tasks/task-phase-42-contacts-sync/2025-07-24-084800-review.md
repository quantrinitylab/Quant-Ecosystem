# Unified contacts subsystem for device-control (v2)

The contacts subsystem adds a unified contact model, in-memory store with CRUD/search/merge, a Capacitor bridge for device sync, CardDAV sync with vCard parsing, and a fuzzy contact resolver. The fix commit addressed all seven v1 issues: `mergeDuplicates` tracks processed IDs, `syncBidirectional` only merges non-empty fields, the vCard parser unfolds continuation lines and splits on first colon, `ContactGroup` has a clarifying comment, the CardDAV conflict strategy is documented, and the resolver returns a distinct `notFound` variant. Watch for: TEL URI scheme prefix still leaks into stored phone numbers (confirmed); resolver's fixed Levenshtein threshold of 2 produces false positives on short names (confirmed); `mergeDuplicates` doesn't merge phone lists from duplicates (confirmed).

## High-level view

The first-colon split fix handles `ADR` values containing colons, but for `TEL;VALUE=uri:tel:+1-555-123` the stored number becomes `tel:+1-555-123` ā€” the URI scheme prefix is preserved because the parser treats everything after the first colon as the value. Digit-stripping in `searchByPhone` masks this during lookups, but `toVCard` serialization emits a double-prefixed URI that other CardDAV clients may reject.

The resolver's Levenshtein threshold is a flat `<= 2` regardless of query length. For a 2-character query like "Al", nearly every short name in the store is within distance 2. The recency tiebreaker only helps when exactly one candidate was recently contacted; otherwise the result is a giant ambiguous set.

`mergeDuplicates` accumulates emails and nicknames from duplicates but does not merge phone numbers or addresses. If contact B (deleted as a duplicate) had a work number alongside the shared mobile number, that work number is silently dropped.

`syncBidirectional` guards against overwriting with empty data, but replaces entire arrays (`phones`, `emails`) when the device has any entries rather than merging. A device contact with one phone number will overwrite a store entry that had three numbers from CardDAV.

## TEL URI prefix preserved in phone values

The first-colon fix splits `TEL;VALUE=uri:tel:+1-555-123` into propPart=`TEL;VALUE=uri` and valuePart=`tel:+1-555-123`. The value is stored as-is:

```typescript
c.phones.push({ number: valuePart, type });
// valuePart = "tel:+1-555-123"
```

`toVCard` serializes this back as `TEL;TYPE=mobile:tel:+1-555-123` ā€” a double-prefixed URI that RFC 6350-compliant clients will reject or misparse. Stripping a leading `tel:` or `sip:` from `valuePart` before storing would fix both the internal representation and the round-trip fidelity.

## Resolver false positives on short names

With threshold 2 applied to full `displayName`:

- Query "Al" (length 2) matches "Bo", "Jo", "Ed" ā€” all within distance 2.
- Query "Jo" matches "Al" (distance 2), "Joe" (distance 1), "Joy" (distance 1), "Jon" (distance 1).

The recency tiebreaker only disambiguates if exactly one candidate was contacted within 7 days. For stores with many first-name-only entries, short queries produce an ambiguous result containing most of the store.

## mergeDuplicates field accumulation is incomplete

The merge loop accumulates `emails` (deduped by address), `nicknames` (via Set), and `lastContacted` (max wins). It does not accumulate `phones`, `addresses`, or `relationships` ā€” the merged contact retains only `unprocessed[0]`'s values for these fields via the initial spread. Since contacts are grouped by shared phone number the "winner" has at least the shared number, but any additional numbers from the deleted duplicate are lost.

## Test coverage gaps remaining

Not tested:

- `TEL;VALUE=uri:tel:...` parsing (URI prefix in phone number)
- `mergeDuplicates` with contacts that have additional phones beyond the shared one
- `syncBidirectional` when existing store contact has more phones than the device contact
- `pushToServer` (still only `syncFromServer` exercised)
