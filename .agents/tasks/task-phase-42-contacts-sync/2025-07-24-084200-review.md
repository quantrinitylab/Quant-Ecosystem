# Unified contacts subsystem for device-control

A new `contacts/` module adds a unified contact model, an in-memory store with search/merge, a Capacitor bridge for native device sync, a CardDAV sync layer with vCard 3.0/4.0 parsing, and a contact resolver with Levenshtein fuzzy matching, nickname/relationship resolution, and recency bias. Watch for: `mergeDuplicates` corrupts contacts that share a phone number with a third contact (confirmed); the Capacitor bridge `syncBidirectional` silently overwrites store data with device data that has less information (confirmed); vCard parsing breaks on folded lines and property parameters with values containing colons (likely).

## High-level view

The contact model defines a `UnifiedContact` interface with typed sub-structures, but `ContactGroup` is exported without being referenced anywhere — `UnifiedContact.groups` is `string[]` rather than `ContactGroup[]`.

`ContactStore`'s merge algorithm groups contacts by normalized phone digits and collapses duplicates, but a contact appearing in multiple phone-number groups can be deleted multiple times, and the "winner" of each group references stale data from the spread at merge start.

`CapacitorContactsBridge` falls back to a no-op stub on web. The bidirectional sync is device-wins-always: it unconditionally overwrites store entries with sparse device data (empty `firstName`, `lastName`, `nicknames`, `relationships`), discarding any local enrichment from CardDAV or user edits.

`CardDAVSync` uses `lastContacted` as a conflict tiebreaker, but `parseVCard` never sets `lastContacted`, so the server version never wins once a contact exists locally. Server-side edits will never propagate without an etag/revision strategy.

`ContactResolver` chains exact, startsWith, nickname, relationship, fuzzy, and recency checks. When zero candidates match, it returns `{ ambiguous: true, options: [] }` rather than a distinct "not found" variant, making caller disambiguation logic error-prone.

<summary>Issues (7)</summary>

1. **mergeDuplicates multi-group corruption** — A contact with two phone numbers appears in two groups. If both groups have duplicates the contact is deleted and mutated in the first pass, then the second pass references stale data. Guard against double-processing by tracking already-merged IDs.
2. **syncBidirectional overwrites local enrichment** — Device contacts imported via Capacitor have empty `firstName`, `lastName`, `nicknames`, `relationships`. `syncBidirectional` unconditionally calls `updateContact` with this sparse data, erasing richer store data. Merge only non-empty fields, or skip update if device contact is less complete.
3. **vCard folded-line handling missing** — RFC 6350 / 2425 requires unfolding lines that begin with a space or tab. The parser does not handle this; multi-line values (long addresses, notes) will be silently truncated or garbled.
4. **vCard colon-in-value truncation** — A property like `TEL;VALUE=uri:tel:+1-555-123` stores `tel:+1-555-123` as the number (the URI prefix leaks through). More importantly, `ADR;TYPE=home:;;123: Main St;...` splits incorrectly because the address value itself contains a colon.
5. **ContactGroup type unused** — `ContactGroup` is defined and exported but `UnifiedContact.groups` is `string[]`. Either the model should reference `ContactGroup[]` or the type should be removed to avoid confusion.
6. **CardDAV conflict resolution always favors existing** — Parsed vCards have no `lastContacted`, so the `>=` comparison always keeps the existing store entry. Server-side updates will never propagate unless a separate version/etag strategy is added.
7. **Resolver returns empty options array** — When no candidates match at all, the resolver returns `{ ambiguous: true, options: [] }`. This is semantically "no match found" but typed as "ambiguous". A distinct "no match" result variant would be clearer.

## mergeDuplicates correctness under multi-phone contacts

The merge algorithm iterates phone-number groups and for each group, spreads `dupes[0]` into `merged`, accumulates data from subsequent entries, deletes them, then writes `merged` back. If a contact has two phone numbers, it appears in two groups. When the first group processes, the contact survives as the merge target. When the second group processes, the same contact may be in `dupes` alongside a different contact. Depending on iteration order:

- If the contact is `dupes[0]` again, the spread copies whatever state was set in the first pass (possibly already merged with data from a third contact that's since been deleted).
- If the contact is `dupes[i>0]`, it gets deleted from the store even though it was the merge winner of the previous group.

Fix: maintain a `Set<string>` of already-merged IDs and skip contacts that have already been processed. Alternatively, build a union-find over contacts and merge each connected component exactly once.

## Capacitor bridge overwrites richer store data

`syncBidirectional` calls `store.updateContact(dc.id, dc)` for every device contact that already exists in the store. The device contact object sets `firstName: ''`, `lastName: ''`, `nicknames: []`, `relationships: []` because the Capacitor plugin only provides `displayName`, phone numbers, and emails. The `updateContact` method does `{ ...existing, ...updates }`, so all those empty fields overwrite whatever richer data was previously stored (e.g., a CardDAV sync that populated `firstName`/`lastName` from the N field, or a user who added relationship tags).

Fix: only merge fields from device data that are non-empty/non-default, or compare `lastContacted` timestamps before deciding update direction.

## vCard parser: folded lines and colon handling

**Folded lines**: vCard 3.0 and 4.0 allow line folding (RFC 6350 section 3.2) where a long logical line is broken with continuation lines starting with a space or tab. The parser's `line.replace(/^\s+/, '')` removes the folding indicator then treats the continuation as an independent line, meaning folded values get split and partially lost. A pre-processing unfold step is needed.

**Colon splitting**: For `TEL;VALUE=uri:tel:+1-555-123`, the `split(':').slice(1).join(':')` produces `tel:+1-555-123` as the stored number — the URI scheme prefix leaks into the phone number. For ADR fields where the value component itself contains a colon (`ADR;TYPE=home:;;123: Main St;Springfield;IL;62704;US`), the initial colon split produces the wrong boundary between property name and value.

## Resolver edge cases

The Levenshtein threshold of 2 is applied against the full `displayName`, which means short names (3-4 characters) can match wildly different contacts. "Al" matches "Jo" with distance 2 (passes the threshold). For names under 4 characters, a proportional threshold (e.g., `distance <= name.length / 3`) would reduce false positives.

## Test coverage gaps

Not tested:

- `mergeDuplicates` with a contact that has multiple phone numbers (the corruption scenario).
- `syncBidirectional` when the store already has richer data than the device.
- vCard parsing with folded lines or unusual property parameters.
- `pushToServer` (only `syncFromServer` is exercised).
