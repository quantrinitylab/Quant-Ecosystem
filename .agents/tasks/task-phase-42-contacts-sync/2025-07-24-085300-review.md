# Unified contacts subsystem for device-control (v3)

A contacts module under `packages/device-control/src/contacts/` delivers a unified contact model, in-memory CRUD store with duplicate merging, a Capacitor bridge for native device sync, CardDAV sync with vCard 3.0/4.0 parsing, and a fuzzy contact resolver. All seven v1 issues and four v2 issues have been addressed: the merge algorithm tracks processed IDs to prevent multi-group corruption, the Capacitor bridge guards against overwriting richer data with sparse device fields, vCard parsing handles line unfolding and first-colon splitting, TEL URI prefixes are stripped, Levenshtein thresholds scale proportionally for short queries, and array fields are union-merged throughout. Watch for: nicknames/relationships in `syncBidirectional` still replace rather than merge (confirmed); `pushToServer` has no test coverage (confirmed).

**Verdict**: APPROVED

## High-level view

The Capacitor bridge's `syncBidirectional` guards every field with a non-empty check and union-merges phones/emails via deduplication filters. One asymmetry remains: `nicknames` and `relationships` still use direct assignment rather than union merge, so a device sync can replace existing lists. The practical impact is low — the Capacitor plugin interface doesn't expose nicknames or relationships — but it breaks the pattern established by phones/emails.

The CardDAV conflict strategy is explicitly documented as local-wins (parsed vCards carry no `lastContacted`, so existing entries always win). The vCard parser correctly unfolds continuation lines, splits on first colon, and strips TEL URI prefixes.

`pushToServer` is the only untested production path. It serializes all contacts to vCard and pushes each to the server — straightforward logic, but a basic test confirming the call pattern would close a coverage gap.

## Nicknames and relationships assignment asymmetry in syncBidirectional

The bridge union-merges phones and emails using deduplication filters:

```typescript
if (dc.phones.length > 0) {
  updates.phones = [
    ...existing.phones,
    ...dc.phones.filter(
      (p) =>
        !existing.phones.some((ep) => ep.number.replace(/\D/g, '') === p.number.replace(/\D/g, '')),
    ),
  ];
}
```

But nicknames and relationships use a simpler pattern:

```typescript
if (dc.nicknames.length > 0) updates.nicknames = dc.nicknames;
if (dc.relationships.length > 0) updates.relationships = dc.relationships;
```

If the store has `nicknames: ['JD', 'Johnny']` and the device has `nicknames: ['JD']`, the sync replaces the list with `['JD']`, losing `'Johnny'`. The fix is trivial — apply `[...new Set([...existing.nicknames, ...dc.nicknames])]`. The practical impact is near-zero because the Capacitor plugin interface doesn't return nicknames or relationships, so `dc.nicknames` will virtually always be empty (guarded by the `length > 0` check).

## Test coverage gap

Not tested: `pushToServer` — the only CardDAV method that serializes contacts back to vCard and calls the client. A single test verifying that `client.pushContact` is called with valid vCard output for each store contact would cover this path.
