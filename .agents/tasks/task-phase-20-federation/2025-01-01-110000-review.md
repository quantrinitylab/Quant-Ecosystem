# Federation package: ActivityPub, Matrix bridge, and moderation

A new `@quant/federation` package implements the ActivityPub protocol layer (actor model with Ed25519 keypairs, HTTP signature sign/verify, WebFinger, NodeInfo, inbox processing for five activity types, outbox publishing with delivery queue), a Matrix bridge (bot + room mapper), and instance-level moderation (blocklist/allowlist). The architecture is in-memory with no persistence layer, meaning this is a protocol skeleton suitable for wiring to storage later. All state lives in Maps and Sets on class instances.

**Watch for:** Inbox accepts activities without verifying HTTP signatures (confirmed). Outbox signs against the first recipient URL only but enqueues the signed payload for all recipients (confirmed). The delivery queue has no persistence or scheduling — retry backoff is calculated but never applied as a delay (confirmed). Actor auto-accepts all Follow requests without consent (confirmed).

## High-level view

The HTTP signature implementation is solid (Ed25519 + RSA-SHA256, digest verification, proper signing string construction) but it's never wired into the inbox path, so remote activities are accepted without cryptographic proof of origin.

The outbox publisher signs the request but uses only the first recipient's URL for the signing string, then delivers the same payload to all recipients. The `(request-target)` component of the signature is wrong for all recipients after the first. Additionally, the signed headers are discarded — only the raw payload is enqueued for delivery.

The delivery queue implements exponential backoff calculation but has no timer or scheduling mechanism. `processNext` is synchronous, so the backoff delay is never enforced — failed jobs are immediately re-enqueued.

The Matrix bridge drops messages silently when a Matrix-side event arrives for an unmapped room. Quant-to-Matrix direction auto-creates rooms on first message, but the reverse path has no equivalent.

Moderation checks the blocklist on inbox receipt via domain extraction from the actor URL. The allowlist-only mode is a good pattern. Domain extraction assumes `actor` is always a string URL, which holds within this codebase but diverges from ActivityPub spec where `actor` can be an object.

<summary>Issues (7)</summary>

1. **Inbox has no signature verification** — The inbox processes activities without calling `verifySignature`. Any remote instance (or attacker) can forge activities. Wire signature verification into `InboxProcessor.process()` or as middleware in `FederationServer.handlePostInbox()`.
2. **Outbox signs against wrong URL for multi-recipient delivery** — `signRequest` is called once with `recipients[0]` as the URL, but the same payload goes to all recipients. Each delivery should get its own signature computed against the actual recipient inbox URL.
3. **Delivery backoff is calculated but never applied** — `calculateBackoff()` returns a delay value but `processNext()` never waits or schedules. A failed job is immediately re-enqueued with no temporal delay, defeating the purpose of exponential backoff.
4. **Auto-accept Follow with no consent mechanism** — Every Follow activity is accepted and an Accept is returned. There's no pending/approval state. Add at least a hook or callback for follow approval.
5. **Actor URL in moderation parsed only as string** — `checkActivity` does `new URL(activity.actor)` but ActivityPub allows actor to be an object. This will throw on object-typed actors.
6. **Matrix bridge silently drops unmapped inbound events** — `onMatrixMessage` does nothing when `getQuantConversation` returns undefined. Consider queueing or returning a status.
7. **WebFinger has no actor existence check** — `WebFingerHandler.handle()` constructs the actor URL without verifying the user actually exists. It will return a valid-looking JRD for any username, leading to 404s when the link is followed.

## Inbox accepts unverified activities

The inbox path in `InboxProcessor.process()` validates the activity schema with Zod and checks moderation, but never verifies that the activity was actually sent by the claimed actor. The `verifySignature` function exists and works (tested with roundtrip), but there's no call site in the inbox or server code.

In `server.ts`, `handlePostInbox` extracts the sender domain from the actor URL and passes it to the processor, but the HTTP-level `Signature` header is never inspected:

```typescript
// server.ts line 89-91
const actorUrl = parsed.data.actor;
const senderDomain = new URL(actorUrl).hostname;
const result = this.inboxProcessor.process(parsed.data, senderDomain);
// ^ no signature verification anywhere in this path
```

Any client that can POST to the inbox endpoint can impersonate any remote actor — forging Follow, Like, Create, Announce, and Delete actions.

## Outbox signature scope mismatch

`OutboxPublisher.publish()` calls `signRequest` once using `recipients[0]` as the target URL:

```typescript
// outbox.ts lines 26-34
signRequest(
  this.privateKey,
  this.keyId,
  'POST',
  recipients[0] ?? 'https://localhost/inbox',
  headers,
  payload,
);
```

The signature includes `(request-target): post /inbox` (or whatever path the first recipient has). The same `payload` string is enqueued for all recipients without re-signing. Any spec-compliant verifier at a different path will reject these deliveries.

The returned signed headers are also discarded — the delivery queue enqueues just the raw `payload` string, so even the first recipient's delivery wouldn't carry the signature headers.

## Delivery queue: backoff without delay

`processNext()` is synchronous and re-enqueues failed jobs immediately:

```typescript
if (!success) {
  const nextAttempt = job.attempt + 1;
  if (nextAttempt < job.maxAttempts) {
    this.queue.push({ ...job, attempt: nextAttempt }); // immediate re-enqueue
  }
}
```

The `calculateBackoff` method returns correct exponential values but is never called by the queue itself. A consumer calling `processNext` in a loop would retry immediately, hammering the remote server. The job schema could carry a `nextAttemptAfter` timestamp, or the queue could expose an async `processWithBackoff` method.

## Follow auto-accept and no Undo/Reject support

The inbox handler automatically accepts every Follow and returns an Accept activity. There's no pending state for manual approval, no Undo Follow handling (unfollowing), and no Reject Follow support. Any remote actor can add themselves as a follower of any local actor without consent.

## WebFinger responds for nonexistent actors

`WebFingerHandler.handle()` constructs and returns a valid JRD for any `acct:user@domain` query without checking whether that user actually exists. A WebFinger probe for `acct:nonexistent@quant.example` returns a 200 with a link to `/users/nonexistent`, which then 404s. The spec expects WebFinger to return 404 for unknown resources.

## Test coverage gaps

The test suite covers happy paths well and exercises real crypto. Notable gaps:

- No test for signature verification in the inbox/server path (because it isn't wired)
- No test that outbox signatures are correct per-recipient
- No test for invalid/malformed actor URLs in moderation's `extractDomain`
- No test for `onMatrixMessage` with an unmapped room (the silent-drop path)
- No integration test for the full flow: actor -> outbox signs -> delivery -> remote inbox verifies
