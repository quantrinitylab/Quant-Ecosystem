# Voice command grammar system for device-control (v2)

A natural-language intent pipeline mapping English and Hindi/Hinglish voice input to structured `DeviceIntent` objects, dispatched through the existing capability registry with tier-based confirmation gating. The v1 issues have been substantially addressed: alias regex is pre-compiled with a length cap, the executor uses `?? 0` for unknown capabilities, sequence execution exposes a `confirmed` option, pattern ordering is documented, and the dead `type` field is removed from `GrammarPattern`.

Watch for:

- **Grammar patterns still re-compile regex per `match()` call** (confirmed) — `CommandGrammar.match` constructs `new RegExp(p.pattern, 'i')` inside the loop on every call, despite the pattern strings being static constants.
- **`CapabilityRegistry.get` accepts only `DeviceCapability`** (confirmed) — calling it with a string cast via `as DeviceCapability` for capabilities like `alarm` or `media` silently passes a value not in the union. The registry always returns `undefined` for these extended capabilities. The `?? 0` fix handles the tier map but doesn't fix the provider-resolution gap.
- **Sequence `confirmed` default still bypasses tier gate for shortcuts** (likely) — `executeShortcut` delegates to `executeSequence` without passing `confirmed`, so it inherits the default `true`. The rationale is documented, but `executeShortcut` doesn't expose the option to callers who might want per-intent confirmation.

## High-level view

The remaining type gap between `DeviceIntent.capability: string` and the `DeviceCapability` union is architectural rather than a bug — the grammar intentionally covers capabilities (alarm, iot, media, search, reminder, email, info, display, app) that don't yet have providers. These patterns will produce "No provider" errors at runtime until the union and registry are extended. The risk is that this gap is invisible to callers who expect `parse() -> execute()` to work for any matched command — the error message doesn't distinguish "not yet implemented" from "provider misconfigured."

`CommandGrammar.match` still allocates a fresh `RegExp` per pattern per call despite the pattern array being a module-level constant. The alias registry was fixed to pre-compile at add-time, but the grammar wasn't given the same treatment. This is 19 allocations per utterance in a latency-sensitive voice path.

`executeShortcut` doesn't thread the new `confirmed` option through to `executeSequence`. A caller wanting per-intent tier checks for macro contents must bypass the method and call `executeSequence` directly.

## Grammar regex compilation on every match call

`CommandGrammar.match` constructs a fresh `RegExp` per pattern per call:

```ts
for (const p of patterns) {
  const regex = new RegExp(p.pattern, 'i');
  if (regex.test(normalized)) { ... }
}
```

The `patterns` array is a module-level constant — its contents never change after module load. Pre-compiling each pattern's regex (storing a `RegExp` alongside the pattern string) would eliminate 19 allocations per parse call. The fix is mechanical: add a `compiled: RegExp` field to each pattern entry, populated at module initialization.

## Provider resolution gap for extended capabilities

The executor's tier-map fix (`?? 0`) prevents a runtime crash for capabilities outside the `DeviceCapability` union. But the subsequent registry lookup still uses the cast:

```ts
const provider = this.registry.get(intent.capability as DeviceCapability);
```

`CapabilityRegistry.get` is typed as `get<T extends DeviceCapability>(capability: T)`, so TypeScript trusts the caller. At runtime, `this.providers` (a `Map<DeviceCapability, ...>`) returns `undefined` for keys like `'alarm'` or `'media'` that were never registered. The executor reports `"No provider for alarm"` — indistinguishable from a misconfigured system. A caller integrating `IntentParser` + `CommandExecutor` sees the grammar match succeed (valid `DeviceIntent`) and execution fail with a generic provider error, with no indication that the capability is structurally unsupported rather than temporarily missing.

If the grammar intentionally covers future capabilities, the error path should distinguish "not yet implemented" from "provider not registered."

## Shortcut confirmation threading

`executeShortcut` doesn't forward the `confirmed` option:

```ts
async executeShortcut(shortcut: CustomShortcut): Promise<ExecutionResult> {
  return this.executeSequence(shortcut.actions, { stopOnFailure: shortcut.stopOnFailure });
}
```

A caller who wants to execute a shortcut with per-intent confirmation cannot use this method — they must decompose the shortcut and call `executeSequence` manually. The documented "pass confirmed=false for per-intent checks" path isn't accessible through the primary shortcut execution method.
