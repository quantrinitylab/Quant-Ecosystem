# Voice command grammar system for device-control

A natural-language intent pipeline that maps English and Hindi/Hinglish voice input to structured `DeviceIntent` objects. The system is layered: user aliases substitute tokens before matching, custom shortcuts (verbal macros) override built-in patterns, and a regex-based grammar handles the rest. A `CommandExecutor` dispatches resolved intents to the capability registry with tier-based confirmation gating for sensitive operations.

Watch for:

- **Capability type widening** (confirmed) â€” the grammar emits capabilities (`alarm`, `iot`, `media`, etc.) that don't exist in the `DeviceCapability` union, so the executor's `as DeviceCapability` cast silently passes unregistered capabilities to the registry, which always returns `undefined` â€” these intents will always fail at runtime with no compile-time warning.
- **Executor skips tier checks for sequences** (confirmed) â€” `executeSequence` passes `confirmed: true` for every intent, meaning tier-3+ actions inside multi-step shortcuts bypass the confirmation gate entirely.
- **ReDoS surface in alias resolution** (likely) â€” user-supplied alias triggers are compiled into `RegExp` at resolve-time with no length or complexity limit enforced.

## High-level view

The grammar is a flat ordered array of regex patterns, matched top-to-bottom with Hindi entries first to prevent false positives from the broader English "call" pattern. This gives correct priority but couples ordering to correctness with no structural enforcement â€” adding a pattern in the wrong position silently breaks existing matches.

The `DeviceIntent` type uses `capability: string` rather than the existing `DeviceCapability` union. This opens a gap at the executor boundary where a type assertion (`as DeviceCapability`) bridges two incompatible type domains with no runtime validation. The grammar covers 8 capabilities that don't exist in the union, making those patterns functionally dead at the executor layer.

Alias resolution compiles a fresh regex per alias per `resolve()` call with no input length bounds. The per-call compilation cost and the O(N\*M) scan over N aliases and M-length text could matter in a latency-sensitive voice pipeline.

The executor's sequence path hard-codes `confirmed: true`, meaning any intent embedded in a shortcut sequence skips confirmation regardless of its tier. A user could bypass the confirmation UX for phone calls or file access by wrapping them in a custom shortcut.

## Capability type boundary between grammar and executor

`DeviceIntent` declares `capability: string`, which lets the grammar emit any capability name â€” the pattern table uses `alarm`, `iot`, `media`, `search`, `reminder`, `email`, `info`, `display`, and `app`, none of which exist in the `DeviceCapability` union type (`phone | sms | contacts | camera | location | sensors | bluetooth | wifi | files | notifications | accessibility`). `CommandExecutor.execute` immediately casts:

```ts
const cap = intent.capability as DeviceCapability;
const tier = CAPABILITY_TIER_MAP[cap];
```

`CAPABILITY_TIER_MAP` is typed as `Record<DeviceCapability, number>`, so looking up a key that isn't in the union returns `undefined` at runtime (the cast suppresses the type check). Every "alarm", "media", or "iot" intent passes the tier check (tier is falsy) and then hits `this.registry.get(cap)` which returns `undefined` â€” failing with "No provider" regardless of system state. The grammar's extended capability coverage is functionally dead unless the union is extended or providers are registered under arbitrary string keys.

## Confirmation bypass in sequence execution

`CommandExecutor.executeSequence` always passes `confirmed: true`:

```ts
const r = await this.execute(intent, true);
```

Any intent executed as part of a shortcut or sequence automatically bypasses the tier-3 gate. A user could create a shortcut like:

```json
{
  "trigger": "do the thing",
  "actions": [{ "capability": "phone", "action": "place", "params": { "target": "..." } }]
}
```

This would place a phone call without confirmation. The original design intent (per FEAT-001: "Tier 3+ actions return `{ requiresConfirmation: true }` unless confirmed flag is passed") suggests the confirmation UX should still apply per-intent. If the bypass is intentional for shortcuts (user explicitly trusts their own macros), it should be documented; if not, the executor should check tier independently for each sequence member.

## Alias resolution and regex construction

`AliasRegistry.resolve` constructs a fresh `RegExp` per alias per call:

```ts
const regex = new RegExp(alias.trigger.replace(/[.*+?^${}()|[\]\\]/g, '\\$&'), 'gi');
result = result.replace(regex, alias.value);
```

No length limit is enforced on triggers. The repeated application of N regex replacements over text of length M gives O(N\*M) worst-case per parse call. Pre-compiling regexes at `add`-time and caching them would avoid per-call compilation overhead and allow validation at registration time (reject triggers that exceed a sane length).

## GrammarPattern `type` field is dead code

The `GrammarPattern.type` field accepts `'exact' | 'regex' | 'keyword'`, but `CommandGrammar.match` treats every pattern identically:

```ts
const regex = new RegExp(p.pattern, 'i');
if (regex.test(normalized)) { ... }
```

The "hindi-navigate" pattern is marked `type: 'exact'` but uses `'^ghar le chal$'` as its pattern string â€” a regex that happens to be anchored. The `type` field suggests the matching strategy should differ by type, which it doesn't. This creates a misleading interface: a consumer extending the grammar might expect `type: 'keyword'` to trigger substring matching rather than full-regex evaluation.

## Test coverage gaps

Not tested:

- An alias trigger containing regex metacharacters (e.g., "c++" as trigger) â€” the escaping logic handles this, but no test verifies it
- A disabled shortcut (`enabled: false`) falling through to grammar match
- `executeShortcut` method directly (only `executeSequence` is covered)
- Grammar patterns with Unicode beyond romanized Hindi (Devanagari script, emoji)
- The behavior when `extract` returns mismatched params for a pattern (no schema validation on extracted params)
