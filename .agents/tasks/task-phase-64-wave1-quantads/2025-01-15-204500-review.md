# QuantAds Next.js App Router conversion with shared-ui adoption

QuantAds moves from plain descriptor-based page exports to a fully wired Next.js App Router application, closely mirroring the QuantSync reference architecture. The change adds a root layout with provider tree (QueryProvider > AppProviders), four route pages (dashboard, campaigns, analytics, audiences), Tailwind + CSS variables for brand tokens and dark mode, and the standard `next.config.mjs` with `transpilePackages`. The API client and types (both pre-existing) remain untouched; pages call into them via TanStack Query hooks.

**Watch for:** The analytics page date-range selector is cosmetic — changing the range triggers a re-fetch but the `listCampaigns` call ignores the `dateRange` parameter entirely, so the UI implies filtering that doesn't happen (confirmed). Mutations on the campaigns page have no `onError` handler, meaning failures are silently swallowed without user feedback (confirmed). The `tsconfig.json` `include` array has redundant entries that signal copy-paste drift (confirmed).

## High-level view

The App Router structure, layout, and provider tree replicate QuantSync's pattern exactly. The globals.css diverges only in `--brand-app-color` (`#10B981` emerald for Ads vs `#1DA1F2` blue for Sync), appropriate for app differentiation.

The campaigns page is the most complete — it wires `useMutation` for pause/resume and delete — but lacks error feedback when those mutations reject. The other three pages are read-only query consumers with proper loading/error/empty states.

There is no inter-page navigation component (no `<Link>` between routes, no sidebar or navbar). QuantSync also lacks one at this stage, but the gap is notable because the command palette commands reference routes (Analytics, Audiences) without actually navigating to them.

1. **Analytics date-range is decorative** — `dateRange` is included in the query key (triggering re-fetch) but never passed to `quantAdsAPI.listCampaigns()`. Either pass it as a filter parameter or remove the selector to avoid misleading the user.
2. **Mutation failures are silent** — `statusMutation` and `deleteMutation` in campaigns/page.tsx have no `onError` callback. Add user-visible feedback (Toast, ErrorState, or inline message) so failures don't disappear.
3. **Command palette actions are no-ops** — All command `action` callbacks are `() => {}`. Wire them to `router.push()` or similar so Cmd+K actually navigates.
4. **Redundant tsconfig include entries** — `"src/app/**/*.ts"` and `"src/app/**/*.tsx"` are already covered by `"src/**/*.ts"` and `"src/**/*.tsx"`. Remove the duplicates to avoid drift.
5. **No inter-page navigation** — Pages exist at distinct routes but there's no way for a user to move between them besides the address bar or a future nav component. Acceptable for a first pass but should be a tracked follow-up.

## Analytics date-range selector gap

The analytics page presents a segmented control for 7d/14d/30d/90d ranges and includes `dateRange` in the TanStack Query key, so changing the range triggers a new network request. But the `queryFn` calls `quantAdsAPI.listCampaigns()` with no arguments — the range is never forwarded. The server returns the same campaign list regardless. This creates a UI where the user sees a loading spinner on range change and gets identical data back, which reads as a broken feature rather than a placeholder.

```tsx
queryKey: ['analytics-overview', dateRange],
queryFn: async () => {
  const response = await quantAdsAPI.listCampaigns(); // dateRange unused
```

The API client exposes a `status` filter on `listCampaigns` but nothing for date range. Either the endpoint needs a date parameter (backend work), or the client-side code should filter by `campaign.createdAt` / `campaign.updatedAt` relative to the selected range.

## Mutation error feedback in campaigns page

Both `statusMutation` and `deleteMutation` define only `onSuccess` (which invalidates queries) and rely on default React Query behavior for errors. Since no `onError` callback or global mutation error handler is configured, a failed pause/delete/resume action produces no user-visible signal — the button returns to idle state and the campaign list doesn't change.

QuantSync also lacks mutation error handlers, so this isn't a divergence from the established pattern. Still a UX gap: destructive actions (delete) should confirm failure visibly.

## Command palette wiring

The five commands (New Campaign, Analytics, Audiences, Creatives, Billing) all have `action: () => {}`. This matches QuantSync's placeholder approach, but since QuantAds now has actual routes for `/campaigns`, `/analytics`, and `/audiences`, the actions could be wired to `router.push()`. Low-priority follow-up.

## tsconfig include redundancy

```json
"src/**/*.ts",
"src/**/*.tsx",
"src/app/**/*.ts",
"src/app/**/*.tsx",
```

The `src/**/*` globs already cover `src/app/**/*`. The duplicates create confusion about whether `src/app` needs special treatment. QuantSync's tsconfig does not have this redundancy.
