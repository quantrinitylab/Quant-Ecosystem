# AI service, Docker infrastructure, and production middleware

This wave adds three production capabilities: a `UnifiedAIService` with provider fallback and mock mode (`@quant/ai`), Docker packaging for four apps with a full docker-compose stack (Postgres/pgvector + Redis + 5 services), and server-core hardening (request logger, graceful shutdown, Kubernetes health probes, initial database migration, seed script). The approach keeps every app deployable without real AI keys by falling back to mock responses — good for local/demo environments but creating a subtle UX gap in production when keys are silently absent.

**Watch for:** graceful-shutdown hook doesn't short-circuit the request lifecycle after sending 503 (confirmed); docker-compose embeds a hardcoded JWT secret that will survive copy-paste into staging (confirmed); `UnifiedAIService` is instantiated per-request in the ask route with no shared state or rate limiting (confirmed); the migration SQL uses `CREATE TABLE IF NOT EXISTS` which will silently no-op on schema drift rather than failing forward (likely).

## High-level view

The `UnifiedAIService` wraps the existing `AIEngine` with a provider-detection layer: when no `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, or `GOOGLE_API_KEY` exists in the environment, every method returns plausible mock data. Real traffic uses a priority-ordered fallback chain (OpenAI > Anthropic > Google). The embedding method always returns mock data even when real providers are available — it makes a text-generation call then discards the result and returns random vectors anyway.

Docker infrastructure follows a consistent multi-stage pattern (deps → builder → runner) with non-root user, `HEALTHCHECK`, and corepack-pinned pnpm. The compose file stands up the full platform on a single bridge network with health-checked Postgres and Redis. The CI matrix was extended to build-test all four new Dockerfiles alongside the existing ws-gateway using a backward-compatible fallback expression for the dockerfile path.

Production middleware lands three Fastify plugins: a structured request logger that emits JSON with userId and response time, a graceful-shutdown plugin that drains connections on SIGTERM/SIGINT, and enhanced health probes (`/healthz`, `/livez`, `/readyz`) with active database and Redis pings. The migration SQL covers the full schema across all apps in a single 0001_init migration.

7. **No rate limiting on /api/ask** — The route has no per-user or global throttle beyond the server-wide rate limiter. An authenticated user can exhaust AI provider budget with rapid calls.

## Graceful shutdown lifecycle gap

The `graceful-shutdown` plugin registers an `onRequest` hook that detects the `isShuttingDown` flag and sends a 503:

```typescript
fastify.addHook('onRequest', async (_request: FastifyRequest, reply: FastifyReply) => {
  if (isShuttingDown) {
    reply.status(503).send({ error: 'Service shutting down' });
  }
});
```

In Fastify's async hook model, calling `reply.send()` does not abort the hook chain — the framework continues into subsequent hooks and the route handler unless the hook returns `reply` (which signals "I've taken ownership of the response"). During the shutdown window, the route handler still executes, tries to send its own response, and triggers a "Reply already sent" error that logs noise and may crash the process before graceful close completes. The fix is `return reply` after the send call.

## Docker secrets and compose hardening

Every service in `docker-compose.yml` receives `JWT_SECRET: dev-jwt-secret-change-in-production-32chars!` as a literal string. The server-core `createApp` validates that the secret is at least 32 characters in production mode, so this specific value _would_ pass that check if someone ran the compose file with `NODE_ENV=production` (which it does — all services set `NODE_ENV: production`). Compose files get committed and copied between environments more often than anyone intends.

The `OPENAI_API_KEY` and `ANTHROPIC_API_KEY` for quantai are correctly templated as `${OPENAI_API_KEY:-}`. The JWT secret should follow the same pattern.

The Dockerfiles copy the full `packages/` directory in the builder stage rather than just the workspace packages each app depends on. This bloats the build context but is functionally correct since turbo filters the build.

## UnifiedAIService design and ask route

The `/api/ask` route instantiates a new `UnifiedAIService()` per request:

```typescript
const service = new UnifiedAIService();
const response = await service.generateText(question, { model, systemPrompt, userId });
```

Each instantiation creates a new `AIEngine`, which re-initializes circuit breakers, cost trackers, and provider health monitors. For production traffic this means no state sharing for circuit-breaker trips — if OpenAI is down, every request rediscovers that independently rather than tripping once and routing to Anthropic.

The mock fallback makes the route always return 200 with plausible content regardless of whether real AI is available. There's no signal to the client indicating "this is a mock response" beyond `model: 'mock-model'` in the payload. If the frontend doesn't check this field, users may believe they're getting real AI answers.

## Embedding and moderation: mock-disguised-as-real

`generateEmbedding` has an `if (!hasAnyProvider()) return mock` branch followed by a "real" branch that:

1. Makes a text-generation call with the prompt "Generate a semantic representation..."
2. Ignores the response
3. Returns `generateMockEmbedding(1536)` anyway

The real branch costs money (a text generation call) but still returns random vectors. If any code downstream uses embeddings for similarity search (the migration creates a pgvector extension), results will be meaningless regardless of provider availability.

`moderateContent` in real mode asks the LLM itself whether content is safe, then searches the response for the substring "unsafe". LLMs can be persuaded to not use trigger words, and qualified responses ("this isn't unsafe per se, but...") pass the check. The mock path always returns safe regardless of input.

## Health probes and readiness semantics

The readiness check (`/readyz`) treats "n/a" (no client configured) as equivalent to "ok", which means a misconfigured deployment (missing Redis URL) will report ready even though features depending on Redis will fail. In production, Redis is required for rate limiting and sessions — a missing Redis URL should fail readiness, not pass it.

## Migration: full-schema init

The 0001_init migration is a 439-line DDL covering 15 tables, 28 enum types, comprehensive indexes, and foreign keys with pgvector support. The `IF NOT EXISTS` pattern on every `CREATE TABLE` means if a previous failed run created some tables but not others (or created tables without their indexes), a retry will skip the existing tables without applying missing indexes or constraints. Prisma's migration engine tracks applied migrations, so this scenario only arises if someone runs the SQL manually outside Prisma, but the risk exists in CI seed scripts.

Seed password hashes use a placeholder string (`$2a$12$placeholder_hash_for_seeding`). If the seed runs against a shared environment, these accounts become unloginable rather than insecure — acceptable for dev, but production seeding needs a separate flow.

## Not tested

- Graceful shutdown under actual SIGTERM (test only verifies registration and normal requests)
- Readiness endpoint with a failing Redis or Postgres connection
- AI service behavior when a provider key exists but is invalid/expired
- Docker builds themselves (CI runs them, but no integration test verifies the container serves traffic)
- The migration SQL against a real Postgres instance
- Request logger output format or presence of userId in log entries
