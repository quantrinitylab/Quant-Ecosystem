# Multimodal capture and grounding for Quant Live pipeline

This change adds camera and screen frame capture with an LLM-backed grounding system to the real-time voice pipeline. Two capture classes (`CameraCapture`, `ScreenCapture`) sample frames at configurable FPS via `setInterval` + `canvas.toDataURL`, feeding base64 JPEG data to a `MultimodalGrounding` class that builds a text-represented multimodal request and streams it through the existing `LiveLLMProvider`. A thin `ContextRetriever` wraps a pluggable search function. The pipeline gains `attachCamera()`, `attachScreenCapture()`, and `getLatestFrame()` methods that store the most recent frame for downstream consumers.

Watch for: frames are text placeholders not actual multimodal content blocks (confirmed); `ScreenCapture` has no FPS cap allowing unbounded CPU usage (confirmed); `autoStart` config is accepted but never honored (confirmed); base64 frame data accumulates without any eviction or size bound (confirmed).

## High-level view

The capture subsystem is a straightforward observer pattern: each capture class owns a `setInterval` that snapshots a `<canvas>` and pushes frames to registered callbacks. The camera enforces a 5fps ceiling; the screen capture does not, despite serving the same role — a caller passing `fps: 30` would get 30 canvas snapshots per second of a full 1280x720 screen.

Multimodal grounding converts frames into text markers (`[image:camera:1706000000]`) embedded in the existing transcript structure rather than passing actual base64 image data to the LLM. This means the LLM never sees the image content; the "multimodal" aspect is a framing protocol that the LLM provider would need to decode externally. Confidence values are hardcoded constants (0.85 with frames, 0.5 without).

The pipeline integration is minimal: `attachCamera`/`attachScreenCapture` register frame callbacks that overwrite a single `latestFrame` field. There's no frame history, no coordination between camera and screen sources, and no trigger connecting grounding to the conversational flow — consumers must call `getLatestFrame()` and wire `MultimodalGrounding` manually.

<summary>Issues (6)</summary>

1. **No FPS cap on ScreenCapture** — `ScreenCapture` accepts arbitrary `fps` values without clamping. Add `Math.min(config.fps ?? 0.5, 5)` as in `CameraCapture`.
2. **Frames are text placeholders, not image data** — `buildContext` emits `[image:camera:timestamp]` strings rather than passing base64 data to the LLM. The provider receives no image bytes. Either pass the data URL in the message or define a multimodal content block structure.
3. **`autoStart` config is dead code** — The `CameraConfig.autoStart` field is stored but never checked in the constructor or `start()`. Either implement auto-start or remove the option from the interface.
4. **Unbounded frame retention in pipeline** — `latestFrame` holds a full base64 JPEG (~50-100KB) indefinitely. If capture stops but the pipeline lives on, stale frame data stays in memory with no TTL or nulling.
5. **Silent failure on capture errors** — Both capture classes swallow all exceptions in `start()` with an empty `catch` block. No error callback, no event, no log. Add an `onError` callback or re-throw so consumers can distinguish permission denial from device absence.
6. **No callback cleanup** — `onFrame` and `onPrivacy` push to arrays with no way to unsubscribe. Repeated attach/detach cycles accumulate dead callbacks.

## Frame capture design and the missing FPS guard

Both capture classes share nearly identical structure: acquire a media stream, create a hidden `<video>` and `<canvas>`, then `setInterval` at `1000/fps` milliseconds to snapshot the canvas as a JPEG data URL. `CameraCapture` clamps the configured FPS to a maximum of 5:

```typescript
fps: Math.min(config.fps ?? 1, 5),
```

`ScreenCapture` does not:

```typescript
fps: config.fps ?? 0.5,
```

A caller passing `fps: 30` gets a 33ms interval calling `canvas.toDataURL('image/jpeg', 0.7)` on a 1280x720 canvas — a blocking JPEG encode on the main thread every 33ms. The fix is trivial (same `Math.min` pattern), but the asymmetry suggests it was overlooked.

## Multimodal grounding: text markers vs. actual image content

`MultimodalGrounding.buildContext()` represents frames as text strings:

```typescript
const imageContent = request.frames.map((f) => `[image:${f.source}:${f.timestamp}]`).join('\n');
```

The actual base64 data from `CaptureFrame.data` is never included in the message sent to the LLM. The `LiveConversationContext` structure uses `TranscriptSegment.text` (a string field) to carry the user message, so there's no structured multimodal content block. The LLM receives:

```
[image:camera:1706000000]
[image:screen:1706000001]

What is on my screen?
```

This works as a protocol if the `LiveLLMProvider` implementation intercepts those markers, fetches frames from a shared store, and converts them to actual multimodal API calls (e.g., Anthropic vision blocks or OpenAI image_url content). But nothing in this diff provides that store or interception logic. As shipped, the LLM treats the markers as opaque text.

The hardcoded confidence values (0.85 with frames, 0.5 without) assert quality rather than measure it — there's no feedback loop from the LLM's response.

## Pipeline integration: last-write-wins single frame

`attachCamera` and `attachScreenCapture` both register callbacks that overwrite the same `latestFrame` field:

```typescript
attachCamera(camera: CameraCapture): void {
  camera.onFrame((frame) => { this.latestFrame = frame; });
}
attachScreenCapture(screen: ScreenCapture): void {
  screen.onFrame((frame) => { this.latestFrame = frame; });
}
```

If both are attached, whichever fires last overwrites the other. There's no per-source latest, no history for the grounding module to pull multiple frames from. `MultimodalGrounding.ground()` accepts an array of frames, but the pipeline only ever exposes one.

The pipeline doesn't wire grounding into `processLLMResponse` — consumers must manually call `getLatestFrame()`, construct a `GroundingRequest`, and invoke `ground()`. The `attachCamera`/`attachScreenCapture` methods create an expectation of integration that doesn't exist yet.

## Silent error handling in capture lifecycle

Both `CameraCapture.start()` and `ScreenCapture.start()` use bare `catch` blocks:

```typescript
} catch {
  this.running = false;
}
```

Permission denied, device not found, constraint unsatisfiable — all land here silently. The consumer's only recourse is polling `isRunning()` after `await start()`. For a voice pipeline that may adaptively enable/disable visual grounding, knowing _why_ capture failed matters — "user denied permission" should trigger a different UX path than "no camera hardware detected."

## Dead `autoStart` configuration

`CameraConfig` defines `autoStart?: boolean` and the constructor stores it, but no code path checks `this.config.autoStart`. The option is part of the public interface but does nothing. Either wire it into the constructor (call `this.start()` if true) or remove it from `CameraConfig`.
