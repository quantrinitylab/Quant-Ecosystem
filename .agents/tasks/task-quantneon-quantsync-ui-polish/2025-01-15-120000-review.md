# UI/UX polish for QuantNeon and QuantSync

Two apps received framer-motion animations (spring-physics via `SpringButton`, `StaggerList`, `PageTransition`, `AnimatePresence`), dark mode Tailwind classes, 44px tap targets, loading skeletons, and responsive layout adjustments. The approach relies heavily on `@quant/shared-ui` motion primitives and CSS custom properties (`--quant-*`) for theming. QuantNeon's core pages (feed, reels, explore) are wired to real hooks; secondary pages retain mock data with UI polish applied. QuantSync's legacy pages receive only a root-level CSS variable swap.

**Watch for:** Nested `<button>` elements in QuantSync's PostCard and feed page create invalid HTML and break assistive technology (confirmed). Dark mode coverage is shallow on QuantSync legacy pages and QuantNeon secondary pages — only the outermost wrapper gets themed while inner elements keep hard-coded `bg-white`, `text-gray-*` (confirmed). SpaceRoom has zero dark mode classes despite being imported into a dark-mode-enabled layout (confirmed).

## High-level view

QuantNeon's primary pages (feed, reels, explore) demonstrate the intended pattern well: real API hooks, proper loading/error/empty states via shared-ui components, responsive containers, and consistent dark-mode token usage. The secondary pages (broadcast, collab, messages, shopping, etc.) receive `PageTransition` wrappers and scattered dark-mode annotations, but still embed large mock datasets inline and have far less thorough theming.

QuantSync's main feed page and component library (`PostCard`, `FeedToggle`, `CommentThread`, `VoteButtons`, `PollCreator`) are substantially polished with animations, dark mode, and accessibility attributes. However, the `SpringButton` component renders its own `<button>`, and several call sites wrap _another_ `<button>` or `<Button>` inside it — producing nested interactive elements that violate HTML semantics and confuse screen readers.

The legacy QuantSync pages receive a one-line dark mode fix (the root container gets `bg-[var(--quant-background)] text-[var(--quant-foreground)]`) but inner elements — sticky headers, cards, modals, inputs, hover states — remain white-only. In dark mode these pages will show white cards and headers on a dark background.

1. **Nested buttons in QuantSync PostCard** — `SpringButton` renders `<motion.button>` but PostCard wraps `<button>` elements inside it, producing `<button><button>`. Use `SpringButton` directly with className/onClick instead of nesting.
2. **Nested buttons in QuantSync feed page** — `<SpringButton><Button>...</Button></SpringButton>` creates double-button nesting. Pass props directly to `SpringButton` or use `Button` alone with motion wrapper.

6. **QuantNeon secondary pages use inline mock data** — Pages like messages, broadcast, collab, shopping embed hundreds of lines of `MOCK_*` arrays. This inflates bundle size and couples sample data to the UI layer.

## Nested interactive elements in QuantSync

In `apps/quantsync/src/components/PostCard.tsx`, the action bar wraps each `<button>` inside a `<SpringButton>`:

```tsx
<SpringButton>
  <button onClick={...} className="flex items-center gap-1 ...">
    ...
  </button>
</SpringButton>
```

`SpringButton` itself renders `<motion.button>`, so the DOM becomes `<button><button>...</button></button>`. This is invalid HTML, breaks keyboard navigation (screen readers expose only the outer button), and produces a browser console warning in most environments.

The same pattern appears in `apps/quantsync/src/app/page.tsx` where `<SpringButton><Button>...</Button></SpringButton>` nests shared-ui's `Button` (which also renders `<button>`) inside a `SpringButton`.

Contrast with QuantNeon's `index.tsx` which uses `SpringButton` correctly — passing `className` and `onClick` directly without nesting another button:

```tsx
<SpringButton className="min-h-[44px] ..." onClick={() => actions.likePost(post.id)}>
  {post.isLiked ? '❤️' : '🤍'}
</SpringButton>
```

This is an accessibility regression. The fix is to either pass props directly to `SpringButton` (the QuantNeon pattern) or replace `SpringButton` with a `motion.div` wrapper that doesn't create an interactive element (confirmed, 2 files affected).

## Dark mode depth disparity

The change applies dark mode at two distinct depths depending on the page:

- **Deep coverage** (feed, explore, reels, PostCard, FeedToggle, CommentThread, VoteButtons, ErrorBoundary, ShoppingCard): Every text element, background, border, hover state, and focus ring gets a `dark:` variant. These pages will render correctly in either theme.
- **Surface-only coverage** (all QuantSync legacy pages, QuantNeon's collab/notes/highlights): Only the outermost container gets `bg-[var(--quant-background)] text-[var(--quant-foreground)]`. Inner children retain `bg-white`, `bg-white/90`, `text-gray-700`, `border-gray-200`, `hover:bg-gray-50` etc. In dark mode, these elements appear as white rectangles floating on a dark background.

Specific example in `apps/quantsync/src/legacy-pages/lists.tsx`:

```tsx
// Root gets dark mode:
<div className="max-w-2xl mx-auto min-h-screen bg-[var(--quant-background)] text-[var(--quant-foreground)]">
  // But sticky header doesn't:
  <header className="sticky top-0 bg-white/90 backdrop-blur border-b z-10">
```

`SpaceRoom.tsx` is a special case — it uses a gradient from `gray-900` which incidentally looks dark, but this isn't a dark-mode-aware implementation. If a light theme were applied, the room would still render dark. More importantly, it has zero `dark:` classes and uses hard-coded white text, meaning it's functionally a single-theme component (confirmed, 0 `dark:` occurrences across 260 lines).

## PostGrid unused `columns` prop

The `PostGridProps` interface still declares `columns?: number` with a default of 3, but the grid template was replaced from dynamic `style={{ gridTemplateColumns: \`repeat(${columns}, 1fr)\` }}`to hard-coded Tailwind`grid-cols-2 sm:grid-cols-3`. The prop is accepted, destructured, and silently ignored. This is dead code that will confuse future consumers expecting configurable column counts (confirmed).

## Mock data coupling in QuantNeon secondary pages

Pages like `messages.tsx` (180+ lines of mock conversations/messages), `broadcast.tsx`, `collab.tsx`, and `shopping.tsx` (120+ lines of mock products) define their test data inline as top-level constants. This creates two concerns:

- **Bundle size**: These arrays ship to production since they're statically imported with no dead-code-elimination boundary.
- **Separation**: The pages can't easily be switched to real API hooks without removing hundreds of lines of mock data that are structurally interwoven with the component.

The feed page (`index.tsx`) demonstrates the clean alternative — no inline mocks, data comes from `useFeed()`, and states (loading/error/empty) are handled via shared-ui components. The secondary pages were polished visually but not architecturally (confirmed).

## Responsive design coverage

The polished pages use `max-w-lg mx-auto px-4` (QuantNeon feed) and `max-w-2xl mx-auto px-4` (QuantSync feed) which provides reasonable 360px–1440px adaptation. The messages page uses a fixed `w-80` sidebar that won't collapse on mobile — there's no responsive breakpoint to hide or overlay it on small screens. This makes the messages page unusable below ~640px viewport width (likely, not verified on a real device but the CSS shows no responsive toggle for the sidebar).
