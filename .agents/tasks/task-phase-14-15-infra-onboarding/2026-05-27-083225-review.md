# Infrastructure observability and onboarding flow system

Phase 14 adds a hand-rolled Prometheus metrics endpoint and request-ID propagation to server-core, alongside SLO burn-rate alert rules, canary analysis rules, a cost dashboard, and operational documentation. Phase 15 introduces `packages/onboarding` with typed state-machine flows for account creation, workspace setup, role-based paths, demo mode, import connectors, AI/privacy/notification preferences, all backed by a 44-test suite.

Watch for:

- The in-process metrics implementation has unbounded memory growth from its route/status cardinality maps (**confirmed**).
- The `/metrics` endpoint is unauthenticated and exposed on the same port as application traffic (**confirmed**).
- Request-ID propagation trusts arbitrary client-supplied header values without validation, enabling log injection (**confirmed**).
- Optional onboarding steps can never be skipped, making flows with `required: false` steps impossible to complete without doing the "optional" work (**confirmed**).

## High-level view

The metrics plugin implements Prometheus exposition from scratch using in-memory Maps keyed by method, route, and status code. Every unique combination grows the map permanently with no cardinality cap or eviction. When `routeOptions.url` is undefined (404s, unmatched paths), it falls back to the raw URL including path parameters, so bot scanners create permanent map entries. The `/metrics` endpoint itself contributes to its own counters on every scrape.

The request-ID plugin trusts whatever string arrives in `X-Request-ID` or `X-Trace-ID` and writes it directly into Pino log context. An attacker can inject control characters or JSON-breaking payloads into structured logs. The surface is reachable without authentication since the plugin is registered before auth.

The onboarding package is a pure-logic library with no I/O. Flows are immutable value objects advanced through typed transitions. The main gap is that `required: false` steps cannot be skipped — the `allDone` check requires every step to be either `completed` or `skipped`, but no function transitions steps to `skipped` status.

<summary>Issues (4)</summary>

1. **Unbounded metric cardinality** — The `metrics.count` and `metrics.duration` maps grow without limit for each unique method/route/status combination. Add a cardinality cap or switch to an established client library (prom-client).
2. **Unauthenticated /metrics endpoint** — `/metrics` is served on the application port without auth. Restrict it to internal traffic (separate port or IP allowlist) or add basic auth.
3. **Log injection via request-ID headers** — `X-Request-ID` and `X-Trace-ID` are written directly into Pino log context without sanitization. Validate format (UUID or alphanumeric+dashes, bounded length) and fall through to `randomUUID()` on mismatch.
4. **No skip-step implementation** — The `OnboardingStepStatus` type includes `'skipped'` and steps have `required: boolean`, but no function allows skipping optional steps. Flows with optional steps can never reach `completedAt` without completing every step. Add a `skipStep` function or change `allDone` to treat `pending` on non-required steps as acceptable.

## Metrics endpoint: cardinality and self-counting

The metrics plugin stores every observed method/route/status triple in a `Map<string, number>`. When `request.routeOptions?.url` is undefined (404s, unmatched paths, or middleware-generated responses), it falls back to `request.url`, which is the raw path including path parameters and query strings. A single bot scanning `/admin`, `/wp-login`, `/phpmyadmin`, etc. will create permanent map entries that are never cleaned up.

```typescript
const route = request.routeOptions?.url ?? request.url;
```

The `/metrics` GET handler records itself in the metrics on every Prometheus scrape (the `onResponse` hook fires for all routes including `/metrics`). At a 15-second scrape interval, this adds 5,760 self-referential data points per day. Not dangerous but adds noise to dashboards. Excluding the metrics route from its own recording would be a one-line fix.

## Unauthenticated /metrics exposure

The auth plugin uses a `requireAuth` decorator that must be explicitly applied per route. The `/metrics` endpoint is a bare `fastify.get` with no `preHandler` guard. If the service is exposed via a public load balancer, `/metrics` leaks route names, request volumes, and latency distributions to unauthenticated callers. This is standard for internal-only services but worth gating if any service-core consumer is publicly reachable.

## Request-ID header trust

```typescript
const requestId =
  (request.headers['x-request-id'] as string | undefined) ?? request.id ?? randomUUID();
```

Any string is accepted and passed to `request.log.child({ request_id: requestId })`. Pino serializes this into structured JSON, so a header value containing `"`, `\n`, or control characters could corrupt log parsers or exploit downstream SIEM ingestion. The fix is trivial: validate against `/^[\w\-]{1,128}$/` and fall through to `randomUUID()` on mismatch.

## Onboarding skip-step gap

The `allDone` check in both account and workspace flows:

```typescript
const allDone = updatedSteps.every((s) => s.status === 'completed' || s.status === 'skipped');
```

This correctly treats `skipped` as a terminal state, but nothing transitions a step into that state. The workspace flow marks "Invite Members" and "Configure Permissions" as `required: false`, implying they can be bypassed. Without a `skipStep(flow, stepId)` function, these flows can never reach `completedAt` unless the user completes every step regardless of the `required` flag.

## Test coverage gaps

Not tested: metric cardinality under high-cardinality load, request-ID validation edge cases (control characters, oversized values), validation schemas for import/AI/privacy/notification modules (these have `validate*` functions with zod schemas but no dedicated test files).
