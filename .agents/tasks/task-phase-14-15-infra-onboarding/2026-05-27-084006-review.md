# Infrastructure observability and onboarding flow system (v2)

Phase 14 adds Prometheus metrics exposition, request-ID propagation, SLO burn-rate alerts, canary analysis rules, cost dashboards, and operational documentation to server-core. Phase 15 introduces `packages/onboarding` with typed state-machine flows for account creation, workspace setup, role-based paths, demo mode, import connectors, and preference configuration. The fix commit addresses all four issues from the v1 review: cardinality cap on metrics, self-counting exclusion, request-ID sanitization, and skip-step implementation for optional onboarding steps.

Watch for: No blocking concerns remain. The fixes are structurally sound and tested.

## High-level view

The metrics plugin now uses registered route patterns (`routeOptions.url`) instead of raw URLs, falls back to `__unmatched__` for 404s, and enforces a 1000-route cardinality cap with an `__overflow__` bucket. The `/metrics` endpoint excludes itself from recording. These changes eliminate the unbounded memory growth vector identified in v1.

Request-ID validation now rejects any header value that doesn't match `^[\w\-]{1,128}$`, falling through to `randomUUID()`. Both `X-Request-ID` and `X-Trace-ID` go through the same validation path, closing the log injection surface. Three new tests cover control characters, oversized values, and trace-ID injection.

The onboarding package gained `skipOptionalStep` functions in both account and workspace flows, with seven new tests covering the full skip-advance-complete lifecycle. The duplication between account and workspace implementations is the only structural concern remaining.

## Cardinality cap edge behavior

The cardinality cap applies to the `knownRoutes` set (tracking `method|route` pairs) rather than the `metrics.count` map directly — status codes still create distinct counter keys within the overflow bucket — but since HTTP status codes are finite, this remains bounded.

The cap never shrinks. If an application dynamically registers and later deregisters routes, those slots remain consumed. For this codebase with static route registration, this is irrelevant. Worth revisiting if a plugin system with dynamic route registration is added later.

## Request-ID bypass via custom genReqId

`request.id` (Fastify's internal ID) is not validated through `SAFE_ID_PATTERN`. Fastify generates these as `req-N` (numeric counter), which passes the pattern. If a custom `genReqId` hook were added elsewhere, it could introduce unsanitized values into log context via the `request.id` fallback path. Not a concern today, but a latent assumption.

## Skip-step duplication

The `skipOptionalStep` function is byte-for-byte identical between `account-onboarding.ts` and `workspace-onboarding.ts`. Both operate on the same `OnboardingFlow` type. A shared `skipOptionalStep(flow: OnboardingFlow, stepId: string)` utility in a common module would prevent future drift if one copy is modified without the other.
