# QuantCalendar: standalone calendar app with AI scheduling (v3)

QuantCalendar adds a full calendar backend to the monorepo: event CRUD with attendee management, hand-rolled RRULE recurrence expansion (daily/weekly/monthly/yearly with BYDAY, BYMONTH, COUNT, UNTIL, EXDATE), multi-user availability intersection, Calendly-style booking links with conflict detection, and six AI-powered features (smart scheduling, buffer insertion, focus blocks, cancellation detection, rescheduling, weekly digest). All seven blocking issues from v1 and two from v2 have been resolved. The overlap-detection queries now use the correct interval predicate (`startTime < rangeEnd AND endTime > rangeStart`) across all conflict-sensitive paths, and the detect-cancellation endpoint requires authentication.

**Watch for:** `listEvents`, `ai-reschedule`, and `ai-weekly-digest` still use `gte/lte` queries that return only fully-contained events, missing events that straddle range boundaries â€” acceptable for display/analytics but worth documenting as intentional (possible). Prompt injection in cancel-detector and reschedule services remains acknowledged-but-unmitigated; the code comments correctly note this is safe while results are only returned to the caller (confirmed).

**Verdict**: APPROVED

## High-level view

The five conflict-detection paths (`confirmBooking`, `getAvailableSlots`, `getAvailability`, `addBufferTime`, `reserveFocusBlocks`) now query with `startTime < rangeEnd AND endTime > rangeStart`, catching events that straddle slot boundaries. The three remaining `gte/lte` queries live in listing/analytics paths (`listEvents`, `ai-reschedule`, `ai-weekly-digest`) where they only miss boundary-straddling events â€” no correctness hazard, but the calendar listing will silently drop events that start before the query window.

Prompt injection in the cancel-detector and reschedule services is explicitly acknowledged in code comments. The safety property is that AI responses are parsed into typed structs and returned to the authenticated caller â€” no server-side mutations are triggered by AI output. That property must hold if the surface is to remain safe.

## Listing query asymmetry with conflict-detection paths

The five conflict-detection services all use:

```typescript
startTime: { lt: dayEnd },
endTime: { gt: dayStart },
```

But three listing/analytics services still use:

```typescript
startTime: { gte: startDate },
endTime: { lte: endDate },
```

In `event.service.ts` line 153, `listEvents` feeds the calendar view. A user with a meeting from 8:00-9:30 who queries events starting at 9:00 won't see that meeting, even though 30 minutes of it falls within the view. The reschedule service has the same gap â€” events spanning midnight on the source day won't be proposed for rescheduling.

This is inconsistent with how the rest of the codebase treats interval overlap, and will produce surprising behavior in the calendar UI. May be intentional if the contract is "events that start and end within the range," but if so it should be documented.

## Prompt injection posture

Both `AICancelDetectorService` and `AIRescheduleService` interpolate user input directly into prompts:

```typescript
// NOTE: messageText is interpolated directly into the prompt without sanitization.
// This is acceptable for now since results are only returned to the caller,
// but should be hardened if automated actions (e.g., auto-cancel) are wired up later.
```

If automated side effects (event deletion, notification dispatch, calendar mutations) are ever wired to the AI output, the injection surface becomes exploitable. The current architecture is safe only because no server-side actions depend on AI responses.
