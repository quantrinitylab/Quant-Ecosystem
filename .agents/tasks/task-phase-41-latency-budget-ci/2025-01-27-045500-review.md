# Latency budget enforcement and CI hard gates

This change introduces a latency budget module for the quant-live voice assistant pipeline, providing programmatic budget definitions (broadband and mobile_4g profiles), assertion helpers with configurable tolerance, a CI runner script, and a test suite exercising budget checks, simulated pipeline latency, memory stability over 100+ turns, and per-minute cost calculations. The approach is purely additive: budget thresholds live in a dedicated `budget/` module exported from the package root, with tests validating the contract from both unit and integration angles.

**Watch for:** The budget system and the existing `LatencyTracker` are entirely disjoint (confirmed) — budget checks operate on manually-assembled `Record<BudgetStage, number>` values and never consume data from `LatencyTracker`, meaning there is no path from live pipeline measurements to automated budget validation. The memory stability test's 5MB threshold assertion is non-deterministic without `--expose-gc` (likely). The cost budget test validates arithmetic in isolation rather than integrating with real pipeline behavior (confirmed).

## High-level view

The budget module defines two network profiles (broadband and mobile_4g) with per-stage latency ceilings for ASR partial, LLM first token, TTS first audio, and total first response. `checkBudget` does a strict comparison; `assertBudget` applies a tolerance multiplier before failing. The two functions duplicate the iteration-and-violation logic rather than one calling the other with tolerance zero, which creates a maintenance seam if stages are added later.

The CI runner (`run-budget-check.ts`) validates hardcoded sample metrics against both profiles and exits non-zero on violation. It does not integrate with any measured latency source — it checks canned values, so its value as a CI gate depends entirely on whether a real metrics source is wired in later.

The memory stability test exercises the full `LivePipeline` over 110 turns with mock providers and asserts that heap growth from turn 10 to turn 110 stays below 5 MB. Without V8's `--expose-gc` flag, the `global.gc?.()` calls are no-ops and the heap measurement includes uncollected garbage, making the test more likely to flake.

The cost budget test defines per-component pricing constants (GPT-4o-mini-level) and validates that a simulated one-minute conversation stays below $0.02. The test is entirely self-contained arithmetic — it exercises its own `calculateCostPerMinute` function without touching the pipeline or any cost-tracking infrastructure.

<summary>Issues (5)</summary>

1. **Budget module disconnected from LatencyTracker** — `checkBudget` / `assertBudget` accept a manually-built `Record<BudgetStage, number>` but the existing `LatencyTracker` tracks stages as `PipelineStage` (`'asr' | 'llm' | 'tts' | 'playback' | 'grounding'`). There is no adapter or shared type. Wire the two together or document the intended integration path.
2. **Duplicated violation logic in assertBudget vs checkBudget** — `assertBudget` re-implements the loop from `checkBudget` instead of calling it. Add a tolerance parameter to `checkBudget` or have `assertBudget` call `checkBudget` and filter by tolerance. This avoids divergence when new stages are added.
3. **generateReport.passed disagrees with results field** — `generateReport` returns `results: result.violations` (strict) but `passed: toleratedViolations.length === 0` (tolerant). A consumer reading `passed: true` might be surprised that `results` contains non-empty violations. Align semantics or rename the field to `strictViolations`.
4. **Memory test relies on global.gc without --expose-gc** — Without that V8 flag, `global.gc` is undefined and the conditional calls silently skip, leaving uncollected garbage in the heap measurement. Add `--expose-gc` to the vitest config for this test, or increase the tolerance to account for GC non-determinism.
5. **CI runner validates canned metrics, not live measurements** — `run-budget-check.ts` always checks the same hardcoded sample. It will never fail in CI unless someone edits the source. Either accept sample metrics via stdin/env or document that it's a template for downstream integration.

## Budget type system vs existing pipeline types

The new `BudgetStage` type defines four string literals: `'asr_partial'`, `'llm_first_token'`, `'tts_first_audio'`, `'total_first_response'`. The existing `PipelineStage` type in `types.ts` defines `'asr' | 'llm' | 'tts' | 'playback' | 'grounding'`. These two type hierarchies never intersect — there is no adapter, no shared enum, and no function that transforms a `LatencyTracker` result into the `Record<BudgetStage, number>` that `checkBudget` expects.

The simulated pipeline tests in `latency-budget.test.ts` work around this by manually recording `Date.now()` deltas and constructing the metrics record by hand. This demonstrates the budget check contract but not the integration path a real CI gate would follow.

## Tolerance inconsistency between checkBudget and assertBudget

`checkBudget` uses strict comparison (`actual > budget`) and returns all violations. `assertBudget` applies `budget * (1 + tolerance)` before comparing. But `assertBudget` re-implements the loop rather than delegating:

```typescript
// latency-budget.ts
if (actual > budget) { violations.push(...) }

// budget-assertions.ts
const threshold = budget * (1 + tolerance);
if (actual > threshold) { violations.push(...) }
```

Adding a new `BudgetStage` requires updating both loops. More subtly, `generateReport` has a third copy of this loop — it calls `checkBudget` for `result.violations` (strict) but then runs its own tolerance loop for the `passed` flag. The `BudgetReport` type returns both `results` (strict violations) and `passed` (tolerance-filtered), creating an API where `passed: true` and `results: [{ stage: 'asr_partial', ... }]` can coexist. Consumers need documentation to understand that `results` uses a different threshold than `passed`.

## Memory stability test: GC and determinism

The test calls `global.gc?.()` before measuring the baseline and after 100 additional turns. In standard Node/vitest execution, `global.gc` is `undefined` because V8 requires `--expose-gc`. The optional chaining means the test silently skips GC and measures heap that includes unreachable objects. Under memory pressure or CI resource limits, garbage accumulation could push the delta above 5 MB and cause a flake.

The vitest config for this package would need `forks: { execArgv: ['--expose-gc'] }` or equivalent. Alternatively, the test could weaken the assertion to a higher threshold (e.g., 10 MB) to absorb GC non-determinism.

## Cost budget: spec vs test threshold discrepancy

The hard gate in the acceptance criteria says `$0.005/min` but the test asserts `$0.02/min` — a 4x discrepancy. If the $0.005 figure is the intended production gate, the test is too lenient. If $0.02 is the real ceiling, the spec should be updated.

## CI runner: main() is never auto-invoked

`run-budget-check.ts` exports a `main()` function that runs hardcoded metrics against both profiles, but `main()` is never called at the module level (no top-level invocation, no `import.meta.main` guard). A `package.json` script entry would need to explicitly import and call `main()` to use this as a CI step.
