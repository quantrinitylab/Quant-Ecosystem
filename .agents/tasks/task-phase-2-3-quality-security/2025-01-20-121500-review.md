# Security hardening fixes for JWT and APNs (v2)

The v1 review flagged five issues. Issue 4 (ESLint rules deliberately disabled) was accepted as a trade-off and is not revisited here. The remaining four have been addressed: the WebSocket empty-secret fallback now uses the same `getJwtSecret()` pattern as the REST middleware, the APNs JWT signature uses `ieee-p1363` encoding, the key-type requirement is documented in the class JSDoc, and tokens are cached for 50 minutes. All four fixes land cleanly.

**Watch for:** The `getJwtSecret()` function is duplicated verbatim in two files (confirmed) — `packages/server/src/middleware/auth.ts` and `apps/quantmeet/backend/routes/ws.ts`. If the validation logic ever needs to change (e.g., adding key rotation or a minimum entropy check), both copies must be updated independently.

## High-level view

The WebSocket endpoint now refuses to operate without a properly configured secret in production, matching the REST API's behavior. The two implementations are identical in logic but separate in source, creating a maintenance coupling that could drift.

The APNs client's JWT generation is corrected: signatures are now encoded in IEEE P1363 format, and a 50-minute cache avoids redundant signing. The key-type constraint (EC P-256) is documented in the class-level JSDoc but not enforced at runtime — a misconfigured RSA key will still produce a token that Apple rejects silently.

## v1 issue resolution

The WebSocket route previously had `const JWT_SECRET = process.env.JWT_SECRET || ''`. It now calls `getJwtSecret()`, which throws in production if the secret is missing or shorter than 32 characters, and falls back to a long dev-only placeholder (`'dev-only-insecure-jwt-secret-not-for-production-use-000'`) in non-production with a console warning. This matches the pattern in `packages/server/src/middleware/auth.ts` character-for-character.

The APNs `generateAuthToken()` method now passes `{ key: this.options.token.key, dsaEncoding: 'ieee-p1363' }` to `sign.sign()`, producing the fixed-width 64-byte signature Apple expects. The v1 issue about DER-encoded output is resolved.

Token caching is implemented with `TOKEN_CACHE_DURATION_MS = 50 * 60 * 1000` (50 minutes). The `generateAuthToken` method checks `this.cachedToken && now < this.tokenExpiry` before regenerating, which avoids redundant EC signing operations at scale.

The key-type concern from v1 is now documented at the class level:

```
NOTE: The token key MUST be an EC P-256 private key in PEM format.
Using any other key type will produce invalid signatures.
```

This is a documentation-only mitigation — no runtime validation exists. If an operator passes an RSA key, `createSign('SHA256')` will produce an RSA-SHA256 signature (valid crypto, wrong algorithm for the ES256 JWT header), and Apple will reject the token with `InvalidProviderToken`. The failure won't be obvious from the error response alone.

## Duplicated secret validation logic

Both `getJwtSecret()` implementations are byte-for-byte identical:

- Throw if `NODE_ENV === 'production'` and secret is missing or under 32 chars
- Warn and return dev fallback if secret is missing in non-production
- Return the secret otherwise

The `getJwtRefreshSecret()` variant exists only in the REST middleware. If the WebSocket service ever needs refresh-token validation, a third copy would appear. This is a minor structural concern — the logic is small and stable — but it violates the monorepo's existing pattern of extracting shared utilities into `@quant/common` or `@quant/security`.
