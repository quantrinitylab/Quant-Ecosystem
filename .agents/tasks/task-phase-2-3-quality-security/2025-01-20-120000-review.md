# Lint setup and security hardening for Quant monorepo

This change makes `pnpm lint` functional across 47 workspace packages (previously zero tasks ran), upgrades vulnerable dependencies (Next.js 14->15, nodemailer 6->8, eliminates node-forge via a custom APNs client), removes hardcoded JWT fallback secrets from production paths, adds production config validation for secret strength, and implements JWT authentication on the QuantMeet WebSocket endpoint. The CI audit step no longer uses `continue-on-error`, making high-vulnerability failures blocking.

**Watch for:** The WebSocket JWT secret falls back to an empty string when `JWT_SECRET` is unset (confirmed), which means jose will still "verify" tokens against an empty key in development — any well-formed JWT with matching issuer/audience will pass. The APNs client's JWT signing uses `createSign('SHA256')` but declares `alg: 'ES256'` in the header (confirmed) — SHA256 with `createSign` produces an ECDSA signature only if the key happens to be an EC key; the code does not validate key type, so a misconfigured PEM will produce an invalid token silently. The ESLint config disables nearly every useful rule (confirmed), making the lint gate pass without catching anything meaningful in new code.

## High-level view

The lint infrastructure is now mechanically functional — every package has a lint script and the root config targets all TypeScript sources — but the rule set disables `no-unused-vars`, `no-explicit-any`, `prefer-const`, and several others, meaning it will only catch syntax-level parse failures rather than actual code quality issues. This is a deliberate trade-off to avoid fixing hundreds of existing violations, but it leaves the gate providing minimal signal.

The secret-handling overhaul takes two different approaches in two locations. `packages/server/src/middleware/auth.ts` uses a well-structured `getJwtSecret()` that throws in production and warns with a long dev fallback. `apps/quantmeet/backend/routes/ws.ts` falls back to empty string via `process.env.JWT_SECRET || ''` and has no production guard, creating an inconsistency where the REST API will refuse to start without a proper secret but the WebSocket server will silently accept any token.

The custom APNs HTTP/2 client replaces @parse/node-apn using Node's built-in http2 and crypto modules. The JWT generation logic declares ES256 in the header but relies on `crypto.createSign('SHA256')` without verifying the key is an EC P-256 key, and outputs DER-encoded signatures where APNs expects IEEE P1363 format — both issues will cause silent token rejection in production.

The dependency upgrades (Next.js 15.5, React 19, nodemailer 8) are reflected in the lock file. The quantchat `[id]/page.tsx` was rewritten to use React 19's `use()` for params unwrapping, which is the correct migration pattern for Next.js 15's async params.

4. **ESLint rules are all disabled** — The lint gate passes but catches almost nothing. Consider enabling at least `no-unused-vars` as a warning and `no-explicit-any` as a warning to provide incremental signal, or add a TODO with a target date for enabling rules.
5. **APNs token not cached** — `generateAuthToken()` is called on every send. APNs recommends reusing tokens for up to 60 minutes. At scale this generates unnecessary CPU work for signing.

## WebSocket JWT auth and the empty-secret gap

The gap is at line 5 of `apps/quantmeet/backend/routes/ws.ts`:

```typescript
const JWT_SECRET = process.env.JWT_SECRET || '';
```

When `JWT_SECRET` is unset, `jose.jwtVerify` receives a zero-length `Uint8Array` as the secret. With HMAC-based algorithms (HS256), jose will still perform verification against this empty key. Any attacker who knows the issuer and audience values (both hardcoded as defaults in this file) can forge a valid token by signing with an empty secret. There is no production guard equivalent to what `packages/server/src/middleware/auth.ts` implements — the WebSocket server will run in production with an empty secret and accept forged tokens.

The server-core `createApp` validates `config.jwtSecret` at startup, but the WebSocket module reads `process.env.JWT_SECRET` directly at module load time, bypassing that check entirely. These two code paths have no shared validation.

## APNs client JWT construction

The APNs client builds its own JWT using Node's `crypto.createSign('SHA256')`. Two issues:

First, ES256 requires the signature in IEEE P1363 fixed-width format (two 32-byte integers concatenated), but `createSign().sign()` outputs DER-encoded ECDSA by default. Apple's APNs service will reject these tokens with `InvalidProviderToken`. The fix is to pass `{ dsaEncoding: 'ieee-p1363' }` as part of the sign options:

```typescript
const signature = sign.sign(
  { key: this.options.token.key, dsaEncoding: 'ieee-p1363' },
  'base64url',
);
```

Second, `createSign('SHA256')` will produce an ECDSA-SHA256 signature only if the provided key is an EC key. If someone passes an RSA key (misconfiguration), `createSign` will produce an RSA-SHA256 signature, which won't match the `ES256` algorithm declared in the JWT header. There's no validation that `this.options.token.key` is actually an EC P-256 private key.

The silent error handler (`session.on('error', () => {})`) means there's no logging when APNs becomes unreachable — failures are invisible until a caller inspects the `ApnsSendResult.failed` array.

## ESLint configuration: functional but toothless

The rule overrides disable:

- `@typescript-eslint/no-unused-vars: 'off'`
- `@typescript-eslint/no-explicit-any: 'off'`
- `@typescript-eslint/no-empty-object-type: 'off'`
- `@typescript-eslint/no-require-imports: 'off'`
- `@typescript-eslint/ban-ts-comment: 'off'`
- `@typescript-eslint/no-unsafe-function-type: 'off'`
- `prefer-const: 'off'`

The `typescript-eslint` recommended config is applied and then immediately gutted. What remains active are mostly parse-level checks that TypeScript itself would catch. The lint gate now passes (which unblocks CI), but it won't catch unused imports, `any` proliferation, or other quality regressions in new code.
