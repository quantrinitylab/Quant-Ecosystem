# Framer-motion primitives and UI/UX pass across 7 apps

This change introduces a shared animation and theming layer in `@quant/shared-ui` built on framer-motion, then wires it into all seven Next.js frontends. The primitives include page transitions (AnimatedPage), stagger lists (AnimatedList), spring-physics buttons and sheets, a command palette overlay, a theme provider with dark mode persistence, and responsive layout hooks. The approach centralizes all motion behind `@quant/brand` spring constants so physics tuning is a single-source change, and each app gets an identical integration pattern: `AppProviders` wrapping `ThemeProvider` + `CommandPaletteUI`, `AnimatedPage` wrapping page content, and expanded `globals.css` for dark mode and responsive breakpoints.

**Watch for:** Missing CSS definitions for CommandPaletteUI and EmptyStateIllustration render them unstyled (confirmed). The `useReducedMotion` hook is exported but never consumed by any motion component, so users who prefer reduced motion still get full animations (confirmed). AnimatedPage exit animations never fire because no app passes a `pageKey` and `AnimatePresence` lives inside the unmounting component (confirmed). The `useBreakpoint` hook fires on every resize event with no throttle (confirmed).

## High-level view

The animation primitives all delegate to `@quant/brand`'s `spring` constants, giving consistent physics across the system. The abstraction is thin — each component is a framer-motion wrapper with props for customization. Any app needing more complex orchestration (route-based exit animations, shared layout transitions) will outgrow these wrappers quickly.

CommandPaletteUI is a fully client-rendered modal with fuzzy search and keyboard navigation. It uses CSS class names (`command-palette-overlay`, `command-palette-container`, etc.) for styling, but no CSS file defining those classes exists anywhere in the repository. The component renders its structure with animations but appears unstyled. The same gap affects EmptyStateIllustration and the `animate-shake` class from `useOptimisticAction`.

The `useBreakpoint` hook attaches a raw `resize` listener with no debounce. On complex pages with many breakpoint-aware consumers, this fires setState on every pixel of a drag-resize.

All seven app integrations follow the same template: `AppProviders` adds theme and command palette, layout gets `suppressHydrationWarning`, page content wraps in `AnimatedPage`. The command actions are all no-ops (`action: () => {}`), which is expected scaffolding but means Cmd+K currently does nothing useful beyond opening the palette.

## Missing component stylesheets

CommandPaletteUI renders a tree of elements with semantic class names: `command-palette-overlay`, `command-palette-backdrop`, `command-palette-container`, `command-palette-input-wrapper`, `command-palette-input`, `command-palette-results`, `command-palette-group`, `command-palette-group-label`, `command-palette-item`, `command-palette-item--active`, `command-palette-highlight`, `command-palette-item-icon`, `command-palette-item-label`, `command-palette-item-shortcut`, `command-palette-footer`, `command-palette-empty`. No CSS defining these selectors exists in the workspace. The framer-motion animations (scale + fade) will fire, but the modal has no positioning, no backdrop color, no layout constraints, no highlight styling.

EmptyStateIllustration has the same issue with `empty-state-illustration`, `empty-state-illustration__svg`, `__title`, `__description`, `__action`. The SVGs use `currentColor` and `var(--app-color)` for brand-adaptive coloring, but without layout CSS the component won't center or space its elements.

`useOptimisticAction` returns `'animate-shake'` as a rollback class. No `@keyframes shake` or Tailwind `animate-shake` extension exists. The visual rollback feedback promised by the hook silently does nothing.

## Accessibility gap: useReducedMotion is decorative

The `useReducedMotion` hook wraps framer-motion's internal hook and is exported from the public API, but no component in this diff consumes it. Framer-motion itself respects `prefers-reduced-motion` at the library level for some animation types, but the explicit spring configs passed via the `transition` prop override that behavior. Components like BottomSheet and PullToRefresh — which rely on drag physics — should check `useReducedMotion()` and fall back to instant/opacity-only transitions.

## AnimatedPage exit animations require route awareness

`AnimatedPage` wraps children in `AnimatePresence mode="wait"` with a `pageKey` prop. No app passes `pageKey` — the default is `undefined`. This means `AnimatePresence` sees the same key across renders and exit animations never fire. More fundamentally, `AnimatePresence` lives _inside_ the page component that Next.js unmounts on navigation, so it gets destroyed before it can run an exit animation. For exit animations to work, `AnimatePresence` needs to live in the layout (above the route slot) and receive `pathname` as the child key.

## Duplicate Cmd+K listeners

Each app's `AppProviders` registers a `keydown` listener that opens the palette. Inside `CommandPaletteUI`, a second listener handles Cmd+K (closing if open) and Escape. Two active listeners for the same shortcut at all times creates a coupling hazard: if the component is later refactored to toggle open/close, both listeners fire and the state oscillates. Consider moving the shortcut ownership entirely into `CommandPaletteUI` with an `onOpen` callback, or removing the component's internal Cmd+K handler.

## Test coverage

Not tested: the 7 app-level `AppProviders` integrations have no tests. `PullToRefresh`'s drag-to-trigger behavior is only tested at the render level (children appear), not the interaction level. The `animate-shake` visual contract is untestable since the CSS doesn't exist.
