# Framer-motion primitives and UI/UX pass across 7 apps (v2)

The fix commit addresses 6 of the 8 v1 issues: CommandPaletteUI and EmptyStateIllustration now use inline styles instead of orphaned class names, `animate-shake` keyframes exist in every app's `globals.css`, `useReducedMotion` is consumed by AnimatedPage and AnimatedList, `useBreakpoint` uses a requestAnimationFrame gate, and the responsive media queries now set distinct `--quant-spacing` and `--quant-font-size-base` values per breakpoint. The AnimatedPage exit-animation limitation is documented via JSDoc. The duplicate Cmd+K listener was resolved by stripping the internal toggle from CommandPaletteUI (it only handles Escape now).

**Watch for:** SpringButton, BottomSheet, PullToRefresh, and SkeletonFade still ignore `useReducedMotion` — only AnimatedPage and AnimatedList were fixed (confirmed). The media query range 361px–413px is uncovered, leaving those viewport widths without responsive custom property overrides (confirmed). The `animate-shake` keyframes are duplicated identically across 7 `globals.css` files rather than defined once in shared-ui (confirmed, low severity).

**Verdict**: APPROVED

## High-level view

The `useReducedMotion` fix covers AnimatedPage and AnimatedList but not the drag-heavy components (BottomSheet, PullToRefresh) that are most likely to trigger vestibular discomfort. Those components pass explicit spring configs via the `transition` prop, which overrides framer-motion's built-in reduced-motion detection.

The responsive media queries now set meaningful per-breakpoint values for spacing and font size, but there's a 361px–413px gap where none of the new variables are defined — this is the most common mobile viewport (iPhone SE at 375px).

The shake keyframes duplication across 7 apps means any future tuning requires 7 synchronized edits.

## Remaining useReducedMotion gap in drag components

The four motion primitives not updated:

- **BottomSheet** slides in from the bottom with spring physics and enables vertical drag. Framer-motion will _not_ automatically skip these because the spring config is passed explicitly via the `transition` prop. A motion-sensitive user interacting with a sheet gets continuous spring-based movement.
- **PullToRefresh** uses `drag="y"` with `dragElastic={0.4}` and a rotating spinner indicator. This is the most motion-intensive primitive and the one most likely to trigger vestibular discomfort.
- **SpringButton** applies `whileTap={{ scale: tapScale }}` and `whileHover={{ scale: hoverScale }}` unconditionally. Low-priority — the visual effect is subtle.
- **SkeletonFade** is opacity-only, generally safe for reduced-motion users. Low priority.

## Media query gap at 361–413px

The breakpoint chain:

```
max-width: 360px     → xs values
min-width: 414px     → sm values
min-width: 768px     → md values
min-width: 1024px    → lg values
min-width: 1440px    → xl values
```

Devices between 361px and 413px (iPhone SE in portrait is 375px) get only the base `:root` values — `--quant-spacing` and `--quant-font-size-base` are undefined at those widths since they're only set inside the media queries. The `:root` block defines `--quant-radius` but not the two new variables. This likely results in those CSS variables being unset (empty string) for the most common mobile viewport width.

## Shake keyframes duplication

Since `useOptimisticAction` is a shared-ui hook, the CSS contract it depends on should live closer to the source. Options: inject the shake via framer-motion's `animate` prop instead of a class (eliminating the CSS dependency entirely), or export a shared CSS file from the package.
