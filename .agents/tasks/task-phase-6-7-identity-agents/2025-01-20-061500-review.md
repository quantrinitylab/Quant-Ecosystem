# Identity, permissions, and agentic execution pipeline (v2)

This change adds workspace-scoped identity/permissions (`@quant/identity-permissions`) and a tiered agent execution pipeline in `@quant/agent-runtime`. The v1 review surfaced six issues; all six have been addressed. The auto-approval is now an explicit opt-in via `autoApprove` option, the PlanGenerator implements relevance matching with a documented fallback, the safety classifier has JSDoc and an `enrichContext` extension point, the tier mapping is corrected, and in-memory growth is documented.

Watch for: the `enrichContext` method is defined but never called from the execution pipeline (confirmed), so the safety classifier context gap is documented but not closed; the `autoApprove: false` path skips execution entirely rather than suspending (confirmed); and `validateArgs` is never invoked during plan execution (confirmed), so required tool parameters are not enforced at runtime.

**Verdict**: APPROVED

## High-level view

The `ExecutionEngineOptions.autoApprove` flag defaults to `true` with clear documentation that production deployments should set it to `false` once async flows exist. When disabled, unapproved steps are skipped rather than suspended â€” there is no resumable approval flow yet, but the seam is in place.

The PlanGenerator now performs word-overlap relevance matching and falls back to including all tools only when zero tools match. The fallback means a vague intent with no keyword overlap still pulls in the entire registry, but workflows control their own tool lists so this is bounded in practice.

The safety classifier's `enrichContext` stub is the documented integration point for identity-permissions signals, but the execution engine does not call it â€” `step.args` is passed directly to `classify`. The hook exists; it just isn't wired into the pipeline yet.

The CSV export in WorkspaceAuditLog still lacks field escaping â€” any field containing a comma produces malformed output.

## enrichContext is an inert extension point

The `ExecutionEngine` at line 92 passes `step.args` directly to the classifier:

```typescript
const safety = this.safetyClassifier.classify(step.toolName, step.args);
```

A one-line change (`this.safetyClassifier.classify(step.toolName, this.safetyClassifier.enrichContext(step.args))`) would close this gap. When identity-permissions integration arrives, only `enrichContext` needs to change. Without this call, integrators must remember to modify the engine â€” the method exists but the pipeline doesn't use it.

## validateArgs is dead code in the execution path

`TypedToolRegistry.validateArgs` checks required parameters and types, but the execution engine never calls it â€” it invokes `tool.handler(step.args)` directly. Since `PlanGenerator.generatePlan` always produces `args: {}`, every tool handler receives an empty object. Once real handlers replace the stubs, missing validation will produce confusing runtime errors rather than clear pre-execution failures.

## CSV export field escaping

The `exportEvents` CSV path joins fields with bare commas without RFC 4180 quoting. If an action name or metadata string contains a comma, the output is unparseable. Low risk while action names are simple identifiers, but a latent bug for any real audit export use case.
