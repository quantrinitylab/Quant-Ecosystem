# Brand system and ecosystem-wide visual identity

A new `@quant/brand` package provides the single source of truth for colors, typography, motion tokens, per-app configurations, SVG logos, and per-app icons across the Quant ecosystem. Seven Next.js apps receive CSS variable integration (`--brand-*` layer), Tailwind brand color utilities, unified favicons, and branded page titles. The `shared-ui` tokens module now imports from `@quant/brand` instead of hardcoding hex values.

Watch for: the CSS generation functions (`generateBrandCSS`, `generateAppCSS`) are exported but never consumed — every app hardcodes its own CSS variables instead (confirmed). Dark-mode tokens in `shared-ui` partially bypass the brand package, using raw hex values that don't correspond to the brand indigo scale (confirmed). No tests exist for the brand package (confirmed).

## High-level view

The brand package is a pure-TypeScript token library: color scales (indigo primary, amber accent, slate neutral, semantic palettes, dark surfaces), typography stacks and scales, spring/easing/duration motion constants, per-app brand configs (14 apps, each with a distinct hue and color), inline SVG icons and logos, and two CSS-generation functions. It ships as `workspace:*` with no runtime dependencies, using TypeScript `as const` for full literal type safety.

App integration follows a layered CSS-variable strategy: each globals.css declares `--brand-primary`, `--brand-primary-hover`, `--brand-accent`, `--brand-accent-hover`, and a per-app `--brand-app-color`, then bridges old `--quant-*` variables to point at the brand layer. Tailwind configs expose these as `brand.*` and `quant.*` utilities. The pattern is identical across all seven apps — the only difference per app is the `--brand-app-color` value.

The token bridge into `shared-ui` is selective: only `primary`, `accent`, and `surface` are imported and used in `lightTokens`/`darkTokens`. The dark-mode primary and accent colors remain hardcoded to different values (blue-400 and violet-400) rather than pulling from the brand indigo scale, which is either an intentional accessibility decision or an incomplete migration.

## Token architecture and the unused CSS generation layer

The brand package exposes two programming interfaces: a TypeScript object tree (`colors`, `typography`, `motion`, `apps`) and a CSS generation API (`generateBrandCSS`, `generateAppCSS`). The TypeScript objects are consumed by `shared-ui/tokens.ts` at build time. The CSS generation functions produce `:root` blocks with complete variable declarations — but nothing in the monorepo calls them.

Each app hardcodes a subset of brand values directly in its globals.css, meaning the package's programmatic token-to-CSS pipeline exists as dead code. If a color changes in `colors.ts`, someone must manually update seven CSS files. The `generateAppCSS` function accepts an `appId` and produces per-app variables from the `apps` registry, which is precisely the manual work each globals.css does by hand. Either a build step should call these functions (e.g. a PostCSS plugin, a code-gen script, or a server component that injects the output), or the functions should be removed to avoid implying a capability that isn't wired up.

## Partial brand adoption in shared-ui dark tokens

In `darkTokens`, the code reverts to hardcoded hex values that belong to a different color family than the brand palette:

```typescript
'--color-primary': '#60A5FA',       // blue-400 — NOT from brand indigo
'--color-primary-hover': '#93C5FD', // blue-300 — NOT from brand indigo
'--color-accent': '#A78BFA',        // violet-400 — NOT from brand amber
```

The brand package's `primary[400]` is `#818CF8` (indigo-400). Light mode uses `primary[600]` correctly; dark mode uses an entirely different hue. The result is that light mode is indigo-branded and dark mode is blue-branded — a visible identity split that users switching between modes will notice.

## Per-app color differentiation gap

The brand package defines 14 app configs, each with a distinct `color` and `hue`. Each app's globals.css sets `--brand-app-color` to the correct per-app hex — but the favicons are all identical: a generic indigo circle with a white "Q". The per-app SVG icons (`quantmailIcon`, `quantcalendarIcon`, etc.) exist in the brand package but aren't referenced in any layout.

Users with multiple Quant apps open see identical tabs. The brand package already has the per-app assets; the wiring to favicons is missing.

## Duplication risk in the CSS variable layer

All seven apps share an identical CSS structure:

```css
:root {
  --brand-primary: #4f46e5;
  --brand-primary-hover: #4338ca;
  --brand-accent: #f59e0b;
  --brand-accent-hover: #d97706;
  --brand-app-color: <per-app>;
  --quant-primary: var(--brand-primary);
  /* ... identical across all apps ... */
}
.dark {
  --quant-background: #0f0f14;
  /* ... */
}
```

The only per-app difference is `--brand-app-color`. The remaining ~25 lines are copied verbatim. If brand-primary changes, all seven files must be updated in lockstep. A shared CSS import or a build step calling `generateBrandCSS()` would collapse this to a single source.

## Color collision: QuantChat and QuantAds

In `packages/brand/src/apps.ts`, both `quantchat` and `quantads` are assigned `color: '#10B981'` and `hue: 160`. The entire purpose of per-app brand configs is to give each product a visually distinct identity — two apps sharing the same color undermines this.

## Bundled but unrelated changes

This diff includes 330 files. The brand-system changes account for approximately 55 files. The remaining ~275 files cover quantads component refactors (JSX migration), new app implementations (quantcalendar, quantdocs, quantdrive, quantmeet), API routes, and package expansions (data-warehouse, dev-platform, voice-first-os, spatial-ui, robotics-bridge, wellbeing, launch-beta, launch-public). Shipping these together makes it impossible to isolate a brand-system regression from a component regression.
