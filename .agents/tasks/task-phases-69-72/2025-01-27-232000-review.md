# Phases 69–72: Zero-Defect Hardening, Universal Tool Layer, Codex/Automate, Phone-Free Agentic Living

Four new packages (error-monitoring, chaos-testing, quant-tools, quant-codex, quant-automate) plus significant additions to quant-live, voice-first-os, shared-ui, and testing. The diff adds ~11,700 lines across 103 files implementing fault injection, 85 typed tool definitions with intent routing, cron-driven automation with durable execution, and voice bridge integration for phone-free workflows. The architecture is internally consistent: each package has clean type exports, standalone tests, and follows the existing monorepo pattern.

**Watch for:** ErrorCapture double-sends events (both queued and immediately dispatched) [confirmed]; `void packageJson` dead code in project-scaffolder [confirmed]; DurableExecutor's `resume()` returns a snapshot but never advances execution state [likely]; NLAutomationBuilder patterns are tightly regex-coupled and won't handle variations well [confirmed]; several test files test simulated behavior rather than real logic paths [confirmed].

## High-level view

The error-monitoring package provides a transport-based error capture system modeled on Sentry. The design is sound but has a double-dispatch bug: `processEvent` both queues events AND immediately sends them to transports, meaning `flush()` re-sends events that were already dispatched. Tests don't catch this because they only verify queue size and event creation, not transport call counts.

The quant-tools package is the most substantial addition: 85 tool definitions across 17 app-specific files with registry, planner, executor, permission engine, undo, audit, and MCP adapter. The types are well-structured with a `PermissionTier` type alias (0|1|2|3) that provides compile-time safety. The MultiStepPlanner and IntentRouter both use simple keyword matching, which is appropriate for a tool-layer foundation. The MCP adapter has proper auth gating with token-to-tier mapping.

The quant-codex and quant-automate packages introduce project scaffolding/orchestration and event-driven automation respectively. Codex templates are extensive (689 lines of inline file content) providing real boilerplate. The cron parser is a well-implemented custom parser that handles step values, ranges, comma lists, and day-of-week normalization. The durable executor stores checkpoints but `resume()` only returns a state snapshot without side effects—it's unclear what the caller does with it.

Phase 72 wires voice transcription to tool execution through a `VoiceToolBridge` that maps keyword patterns to tool routes. The bridge has its own permission tier and app-gating independent of the quant-tools `PermissionEngine`, creating two parallel permission systems. The daily brief system aggregates items from registered sources and generates voice scripts with duration estimation.

## ErrorCapture double-dispatch and queue semantics

In `error-capture.ts`, the `processEvent` method both pushes events to `eventQueue` (line ~130) and then immediately iterates `this.transports` calling `transport.send(event)` (lines ~136-140). When `flush()` is later called, it processes the same events from `eventQueue` and sends them again via the same transports. This means every captured event is sent to transports twice.

```typescript
// processEvent does BOTH:
this.eventQueue.push(event); // queued for flush()
for (const transport of this.transports) {
  transport.send(event).catch(() => {}); // also sent immediately
}
```

The fix is either: remove the immediate sends (defer to `flush()`), or don't queue events that were already sent. The current tests don't catch this because they verify `getQueueSize()` and event metadata but never assert on transport call count.

## Dead code in ProjectScaffolder

In `packages/quant-codex/src/scaffolder/project-scaffolder.ts`:

```typescript
const packageJson = {
  name: projectName,
  version: '1.0.0',
  private: true,
  scripts: template.scripts,
  dependencies: template.dependencies,
};
filesCreated.push(`${projectName}/package.json`);
void packageJson; // explicitly silenced but never written
```

The `void packageJson` statement explicitly silences the "unused variable" warning. The scaffolder claims to create a `package.json` file (adds it to `filesCreated`) but never actually writes the content anywhere. This is internally inconsistent—either the scaffolder should be in-memory-only (and not list package.json as created) or it should store the content.

## DurableExecutor: checkpoint without resume logic

`DurableExecutor` supports `start()`, `checkpoint()`, and `resume()`. However `resume()` only returns a copy of the state object:

```typescript
resume(automationId: string): DurableState | null {
  const state = this.states.get(automationId);
  if (!state || !state.resumable) return null;
  return { ...state };
}
```

There's no mechanism to re-execute pending steps from the last checkpoint. The caller gets a state snapshot but would need to re-implement step iteration themselves. This means "durable execution" is really "durable state storage"—the execution part is left to the caller with no helper. The `FlowController` and `ActionExecutor` don't integrate with `DurableExecutor` at all.

## Parallel permission systems between quant-live and quant-tools

`VoiceToolBridge` in quant-live maintains its own route-based tier system:

```typescript
interface ToolRoute {
  toolId: string;
  appId: string;
  tier: number; // 1 or 2
  // ...
}
```

Meanwhile `PermissionEngine` in quant-tools evaluates permissions against `ToolDefinition.permissionTier`. These are logically the same concept but maintained separately with hardcoded values in VoiceToolBridge (e.g., `quantmail.send` is tier 2 in the bridge but tier 1 in mail-tools.ts). If the tool's permission tier is updated in quant-tools, VoiceToolBridge won't know about it.

## ActionExecutor simulates all executions

In `packages/quant-automate/src/execution/action-executor.ts`:

```typescript
async execute(action: AutomationAction, context: Record<string, unknown>): Promise<ExecutionResult> {
  const success = !context['__force_failure'];  // magic key
  // ...
}
```

The executor never calls any real tool. It returns success unless a magic context key is set. This makes the `FlowController` tests exercise the flow logic (loops, retries, branches) against a stub, which is valid for unit testing, but there's no integration point where `ActionExecutor` would delegate to `ToolExecutor` from quant-tools. The two systems are not wired together.

## ErrorBoundary.setupGlobalHandler is effectively dead

```typescript
setupGlobalHandler(): () => void {
  if (!this.options.captureUnhandled) return () => {};
  const handler = (event: { error?: Error; reason?: unknown }) => { ... };
  return () => { void handler; };  // cleanup does nothing, handler never attached
}
```

The handler function is created but never registered on `window.addEventListener` or `process.on`. The returned cleanup function also does nothing. This is presumably a placeholder for a browser/Node environment, but as-is it's dead code that gives a false sense of global error capture.

## Test quality observations

Tests across all packages are structurally sound—they instantiate classes, exercise methods, and assert results. The chaos-testing tests are particularly good: they verify health checker timeouts, fault injection clamping, circuit breaker trip counts, and resilience report generation with real async operations.

However, several test suites test simulated behavior:

- `quant-automate` executor tests always succeed because `ActionExecutor.execute()` is a simulation
- `quant-codex` build orchestrator tests verify that tasks are decomposed and agents assigned but the "build" is just `await Promise.resolve()` per task
- `quant-live` voice bridge tests verify keyword routing but the "execution" is just object construction

This is appropriate for Phase 70-72 (establishing the architecture and contracts), but means reverting the implementation to a no-op wouldn't break most tests—they're testing the routing/classification logic, not the execution logic.

## Cron parser correctness

The `CronParser` is one of the more algorithmically substantive pieces. It correctly handles:

- Wildcard, step (`*/5`), range (`1-5`), and comma-list fields
- Day-of-week normalization (both 0 and 7 = Sunday)
- `getNextRun` with minute-by-minute search bounded at 366 days
- Human-readable `describe()` output

The minute-by-minute iteration in `getNextRun` is O(n) worst case with n up to 527,040 iterations. For expressions like `0 0 29 2 *` (Feb 29) this could be slow, though bounded.

## ESLint security rules incomplete

The eslint.config.mjs adds `no-console` (warn), `no-eval` (error), and `no-implied-eval` (error). The spec requested `no-new-Function` as well (prevents `new Function('...')` as an eval equivalent), but it's not present.
