# Domain services for all 9 Quant-Ecosystem apps

Phase 6 adds backend service layers for every app in the ecosystem: QuantChat (messaging, encryption, delivery receipts, disappearing messages, media, typing), QuantMail (email CRUD, AI-powered compose/summarize/phishing-detection), QuantAI (sessions, chat with streaming, tool calling, agent marketplace, usage tracking), QuantSync (posts, feed ranking, communities), QuantAds (campaigns, ad serving, analytics), QuantTube (videos, channels, watch history), QuantNeon (photos, stories, filters), QuantEdits (projects, assets, export jobs), and QuantMax (dating profiles, matching, swipes). Each app follows the same structural pattern: services accept a Prisma client (or typed equivalent) via constructor, routes use Fastify with Zod validation, tests mock Prisma with vitest fns. 198 tests total.

**Watch for:** Calculator tool uses `Function()` constructor for eval, which is a code-injection vector even with the regex guard (confirmed). The E2E encryption service uses HMAC as a stand-in for real Diffie-Hellman key exchange, which provides no actual cryptographic security (confirmed). Several services use in-memory state (media metadata, export jobs, typing timers, disappearing message timers) that won't survive restarts or scale across instances (confirmed). The `GET /encryption/keys/:userId` endpoint returns prekey bundles without any authentication check (confirmed). Feed service ignores the userId parameter, returning global public posts instead of a personalized feed (confirmed).

## High-level view

The structural pattern is consistent and well-factored: every service takes its dependencies through the constructor, routes handle auth extraction and Zod validation before delegating, and tests exercise the happy path plus key error branches. The separation makes each service independently testable. The PrismaClient type placeholder (`type PrismaClient = any`) in 5 apps (QuantSync, QuantTube, QuantNeon, QuantAds, QuantMax) erases all type safety at the service boundary while the other 4 apps import the real PrismaClient type, creating an inconsistency in how much the compiler can catch.

The tool-calling framework in QuantAI ships a calculator tool that evaluates arbitrary expressions via `Function()`. The regex allowlist (`/^[\d\s+\-*/().]+$/`) blocks obvious letter-based payloads but `Function()` still creates an execution context that could be abused if the regex is ever relaxed or bypassed via unicode normalization.

QuantChat's encryption service implements a simplified "X3DH-style" key exchange using HMAC chaining instead of actual elliptic-curve Diffie-Hellman. The result is a shared secret that anyone who reads the stored keys can reconstruct. The `GET /encryption/keys/:userId` route compounds this by serving any user's prekey bundle without authentication, so a caller can fetch bundles and compute "shared secrets" for arbitrary sessions.

In-memory state is used in four places: `MediaService` stores upload metadata in a Map, `ExportService` stores job queues in an array, `TypingService` tracks active typers with timers, and `DisappearingService` schedules message expiry with `setTimeout`. All four lose state on process restart and can't distribute across multiple instances. The disappearing service does include a `processExpiredMessages()` batch sweep as a fallback, which is good, but the media and export services have no persistence fallback at all.

The `likePost`, `likePhoto`, and `incrementView` methods use non-atomic read-then-write patterns (`post.likeCount + 1`) instead of Prisma's `{ increment: 1 }`. Under concurrent load, two requests reading the same count will both write `count + 1`, losing one increment. QuantAds uses `{ increment: 1 }` correctly for impressions and clicks, showing the intended pattern.

Route handlers instantiate new service instances on every request (`new MessageService(prisma as never)`). This is fine for stateless Prisma-backed services, but means `ChatService` in QuantAI reconstructs the engine binding per-request. The `as never` casts suppress type checking entirely at the boundary between route and service.

<summary>Issues (8)</summary>

1. **Function() eval in calculator tool** — The `Function("use strict"; return (${expression}))()` call is a code-injection surface. Replace with a safe math parser library (e.g., mathjs `evaluate` in sandbox mode) or at minimum validate the parsed AST.
2. **Unauthenticated prekey bundle endpoint** — `GET /encryption/keys/:userId` serves cryptographic key material for any user without requiring the caller to be authenticated. Add auth middleware.
3. **HMAC-based "key exchange" provides no forward secrecy** — The encryption service uses HMAC with stored keys instead of real ECDH. Document this explicitly as a stub that must not be used for actual encryption, or replace with proper X3DH using `crypto.diffieHellman`.
4. **Non-atomic counter increments** — `likePost`, `likePhoto`, `incrementView` read-then-write counts. Use Prisma's `{ increment: 1 }` atomic update to avoid lost writes under concurrency.
5. **In-memory media metadata has no persistence** — `MediaService` stores upload records in a Map. If the process restarts after a client receives an upload URL but before they fetch metadata, the record is lost. Back with Prisma or Redis.
6. **In-memory export queue has no persistence** — Same issue for `ExportService`. Job state is lost on restart with no recovery path.
7. **Feed personalization is stubbed** — `FeedService.getFeed()` accepts a userId parameter but returns all public posts globally. The comment says "from followed users" but the query has no follow-based filter.
8. **PrismaClient = any in 5 apps** — The type placeholder erases compile-time safety for all service method signatures in QuantSync, QuantTube, QuantNeon, QuantAds, and QuantMax. Consider defining narrow interfaces (like QuantAI does with `SessionPrismaClient`) for type safety without the full Prisma import.

## Calculator tool code injection

In `apps/quantai/backend/services/tool.service.ts`, the built-in calculator tool:

```typescript
handler: async (args) => {
  const expression = args['expression'] as string;
  if (!/^[\d\s+\-*/().]+$/.test(expression)) {
    throw new Error('Invalid math expression');
  }
  const result = Function(`"use strict"; return (${expression})`)();
  return String(result);
},
```

The regex allows parentheses and periods, which is sufficient for math but the use of `Function()` constructor is on most security linting deny-lists because it's equivalent to `eval`. If the regex is ever modified (e.g., to support scientific notation with `e`, or named constants), the door opens to arbitrary code execution. A safer approach is a purpose-built expression parser. The current regex is tight enough to prevent immediate exploitation, but the pattern is fragile.

## Encryption service cryptographic design

The `EncryptionService.establishSession()` method performs:

```typescript
const ephemeralKey = crypto.randomBytes(32).toString('hex');
const dh1 = crypto
  .createHmac('sha256', ephemeralKey)
  .update(responderBundle.identityKey)
  .digest('hex');
const dh2 = crypto
  .createHmac('sha256', ephemeralKey)
  .update(responderBundle.signedPreKey)
  .digest('hex');
```

This is HMAC keyed by an ephemeral value over the _public_ keys stored in the bundle. Real X3DH derives shared secrets from private-key DH operations. Here, anyone who knows the ephemeral key (which is generated server-side and used to derive the `rootKey`) can reconstruct the session key. The "signature verification" in `uploadPreKeyBundle` is also HMAC-based using the identity key as the HMAC key, which means the "signature" is just a self-MAC, not a cryptographic signature.

The `GET /encryption/keys/:userId` route serves bundles without authentication:

```typescript
fastify.get<{ Params: { userId: string } }>('/keys/:userId', async (request, reply) => {
  const bundle = await service.getPreKeyBundle(request.params.userId);
  return reply.send({ success: true, data: bundle });
});
```

No `auth?.userId` check. Any caller can enumerate and retrieve all users' key bundles.

## Non-atomic counter pattern

In `PostService.likePost()`:

```typescript
return this.prisma.post.update({
  where: { id: postId },
  data: { likeCount: post.likeCount + 1 },
});
```

Compared to the correct pattern used in `AdServingService.recordImpression()`:

```typescript
await this.prisma.campaign.update({
  where: { id: adSet.campaignId },
  data: { totalImpressions: { increment: 1 } },
});
```

The same issue affects `PhotoService.likePhoto()` in QuantNeon and `VideoService.incrementView()` in QuantTube. Under concurrent requests, the read-then-write causes lost updates.

## In-memory state and horizontal scaling

Four services maintain process-local state:

- `MediaService` — `private metadata: Map<string, MediaMetadata>` stores all upload records. Production note in comments says it's simulated, but there's no Prisma fallback.
- `ExportService` — `private exports: ExportJob[]` and `private idCounter = 0`. Sequential IDs will collide across instances.
- `DisappearingService` — `private scheduledJobs: Map<string, NodeJS.Timeout>`. The `processExpiredMessages()` batch method provides a safety net, but scheduled timers are lost on crash.
- `TypingService` — `private typingUsers: Map<string, Map<string, TypingState>>`. Ephemeral by nature but won't work in multi-instance deployments; Redis pub/sub would be needed.

The `TypingService` and `DisappearingService` both include `destroy()` methods for cleanup, which is good lifecycle management.

## Route-level service instantiation

Every route handler creates a new service instance per request:

```typescript
const prisma = (fastify as unknown as { prisma: unknown }).prisma;
const service = new MessageService(prisma as never);
```

The double type cast (`unknown` then `never`) discards all type information. For Prisma-backed services this is functionally harmless (services are stateless), but for `ChatService` it means the AI engine reference is also cast through `never`, hiding any interface mismatches until runtime. The encryption route takes a different approach — instantiating the service once at plugin registration time — which is cleaner and avoids repeated allocations.

## Feed service personalization gap

`FeedService.getFeed()` accepts a `_userId` parameter (prefixed with underscore, indicating unused) and queries all public posts without filtering by followed users:

```typescript
async getFeed(_userId: string, options: PaginationOptions = {}): Promise<PaginatedResult<Post>> {
  // Get posts from followed users sorted by recency and engagement
  const [data, total] = await Promise.all([
    this.prisma.post.findMany({
      where: { deletedAt: null, visibility: 'PUBLIC' },
      ...
    }),
```

The comment describes the intent ("from followed users") but the implementation is a global query. This means the feed endpoint returns the same results for every user.

## Test coverage shape

198 tests across 9 apps is solid for a service-layer-only change. The tests exercise ownership checks, pagination, error paths (not-found, forbidden), and happy-path CRUD. Coverage gaps:

- No route-level integration tests (request → response cycle through Fastify)
- No tests for `DisappearingService.processExpiredMessages()` or timer-based expiry
- No tests for `TypingService` timer auto-clear behavior
- No tests for `MediaService` content-type validation rejection
- Streaming (`ChatService.streamMessage`) is tested only for the happy path; error mid-stream is untested
- `ExportService` has no test for the `cancelExport` "not QUEUED" rejection path

The test pattern of mocking Prisma with `vi.fn()` and casting `as never` means tests verify service logic but can't catch schema mismatches between what services pass to Prisma and what Prisma actually expects.
