# Twilio SMS provider with message store and agent (v2)

This change adds SMS capability to the device-control package: a dual-mode Twilio provider, in-memory message store, webhook handler with signature validation, and a message agent for intent dispatch. The v1 review raised five issues; two were fixed (MessageStatus validation now guards against invalid strings, mock-mode `onIncomingSMS` no longer retains unreachable callbacks). Three were explicitly not fixed as intentional parity with existing patterns (permission model, unbounded store, auth token exposure).

**Watch for:** Silent fallback to mock behavior when `initialize()` is never called in live mode — a production caller that skips init gets fake SIDs with no error (confirmed). The `sendSMS` live-mode path does not validate the `to` parameter format, allowing arbitrary strings to reach the Twilio API (confirmed, low severity since Twilio rejects them server-side).

**Verdict**: APPROVED

## High-level view

The two confirmed bugs from v1 are cleanly resolved. Status validation uses a `VALID_STATUSES` array checked before the cast, and the mock-mode unsubscribe no longer retains a dead reference. The remaining unfixed items (prompt-state bypass, unbounded store, token exposure) are deliberate architectural choices consistent with the call-provider subsystem and not regressions introduced by this PR.

The one remaining concern of substance is the silent fallback: `sendSMS` in live mode with a null client (because `initialize()` was never awaited) silently produces mock SIDs. This is a fail-open path that could mask integration failures in production. It's a pre-existing pattern inherited from `TwilioPhoneProvider`, but SMS messages that appear sent but never reach the carrier are harder to debug than calls that never connect.

## Silent fallback to mock in live mode

In `twilio-sms-provider.ts`, the `sendSMS` method:

```typescript
async sendSMS(to: string, body: string): Promise<string> {
  if (this.liveMode && this.client && this.config) {
    // live path
  }
  // mock path — always reached if client is null
  const sid = `SM_mock_${++this.msgCounter}`;
  ...
}
```

When constructed with credentials (live mode) but without an injected client, `this.client` stays null until `initialize()` is called. If a caller creates the provider, sets credentials via env vars, but forgets to await `initialize()`, every `sendSMS` call silently returns a mock SID. The message is stored locally with status `sent` but never reaches Twilio. Unlike a thrown error, this failure mode is invisible to the caller — `success: true` propagates all the way up through the message agent.

This is the same pattern as `TwilioPhoneProvider.placeCall()`, so it's not a regression. But SMS has a subtler failure signature than calls (no real-time audio feedback), making silent mock-mode more dangerous. A single-line guard at the top of `sendSMS` — `if (this.liveMode && !this.client) throw new Error('Provider not initialized')` — would convert this from a silent data loss path to a loud failure.
