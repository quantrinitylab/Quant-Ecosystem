# Twilio SMS provider with message store and agent

This change adds a full SMS capability to the device-control package: a dual-mode Twilio provider (real client when credentials are supplied, mock otherwise), an in-memory message store with CRUD/search/thread operations, a webhook handler for inbound messages and delivery status callbacks, and a message agent that dispatches send/read/reply intents with permission gating and contact resolution. The implementation mirrors the existing call-agent/phone-provider pattern. Twenty-six new tests across four files cover the happy paths and key error conditions.

**Watch for:** The webhook handler accepts any `MessageStatus` string from Twilio and casts it through `as never` to bypass type narrowing, which means a malformed status callback silently corrupts store state (confirmed). The permission check only gates on "denied" state, meaning the default "prompt" state allows SMS actions without explicit user consent (confirmed). The mock-mode `onIncomingSMS` leaks an unreachable callback array on every subscription (confirmed, low severity).

## High-level view

The provider follows the established dual-mode pattern: constructor sniffs for credentials, picks live vs mock, and exposes the same `SMSCapability` interface either way. The webhook handler validates Twilio signatures before mutating store state, but it trusts the `MessageStatus` field without constraining it to the known union type.

The message agent is a thin intent dispatcher that resolves contact names to numbers, checks permissions, and delegates to the SMS provider. Its permission model is permissive: only explicit `denied` blocks actions; the default `prompt` state is treated as allowed. This is consistent with `PermissionManager.request()` which auto-grants on first prompt, but it means any code path that creates a `MessageAgent` without first requesting permission will silently succeed.

The message store is purely in-memory with no eviction or size limit, suitable for development but a concern for any long-running process.

<summary>Issues (5)</summary>

1. **Unvalidated MessageStatus cast** ã `handleStatusUpdate` casts `payload.MessageStatus` through `as never` without checking it against the `MessageStatus` union. A malformed Twilio callback (or attacker) can inject arbitrary status strings into the store. Validate against the known set before calling `updateStatus`.
2. **Prompt-state permission bypass** ã The agent only blocks on `denied`; the `prompt` state (default) allows all SMS actions without user consent. Either gate on `granted` explicitly, or call `permissionManager.request('sms')` to trigger the consent flow before proceeding.
3. **Mock-mode onIncomingSMS leak** ã Each call to `onIncomingSMS` in mock mode allocates a new `callbacks` array, pushes the callback, and returns a no-op unsubscribe. The callback is retained by the array's closure but can never fire or be cleaned up. Use a shared instance array or don't retain the callback at all.
4. **Unbounded in-memory store** ã `MessageStore` has no TTL, size cap, or eviction policy. In a long-running process, memory grows without bound. Add a max-size or TTL-based eviction if this will run beyond tests.
5. **Auth token in webhook handler config** ã `SMSWebhookHandler` stores the `authToken` directly on the instance, and the handler is exposed via `getWebhookHandler()` on the provider. Any code that receives the handler has transitive access to the token. Consider validating via a callback or keeping the token in a closure rather than as an instance property.

## Unvalidated status cast in webhook handler

The `handleStatusUpdate` method in `sms-webhook-handler.ts` takes `payload.MessageStatus` (a `string | undefined` from the `SMSWebhookPayload` type) and passes it directly to `store.updateStatus` via `as never`:

```typescript
if (payload.MessageStatus) {
  this.store.updateStatus(payload.MessageSid, payload.MessageStatus as never);
}
```

`MessageStore.updateStatus` accepts `MessageStatus` (the union `'queued' | 'sent' | 'delivered' | 'failed' | 'received'`), but the cast circumvents that constraint entirely. If Twilio sends an unexpected status value (or if an attacker crafts a request that passes signature validation), the store will contain an SMSMessage with an invalid `status` field. Downstream code that matches on `msg.status` may fall through without handling the invalid value.

Validate against the known set before calling `updateStatus`, and silently ignore or log unknown statuses.

## Permission model defaults to allow

In `message-agent.ts`:

```typescript
const permState = this.permissionManager.getState('sms');
if (permState === 'denied') {
  return { success: false, error: 'SMS permission denied' };
}
```

The `PermissionManager` defaults to `'prompt'` for unset capabilities. The agent treats `prompt` as allowed. The existing `PermissionManager.request()` auto-promotes `prompt` to `granted`, so in a flow where `request()` is called first this works. But the agent doesn't call `request()` itself, and nothing in the call chain enforces that the caller did either. A caller that skips the consent flow gets full SMS access.

If the agent is always fronted by a UI that gates on `request()`, this is acceptable. If the agent is directly reachable from an automation layer, it's a privilege escalation path.

## Mock-mode callback leak in TwilioSMSProvider

In `twilio-sms-provider.ts`, the mock-mode path of `onIncomingSMS`:

```typescript
onIncomingSMS(cb: (msg: SmsMessage) => void): () => void {
  if (this.webhookHandler) {
    return this.webhookHandler.onIncoming(cb);
  }
  const callbacks: Array<(msg: SmsMessage) => void> = [];
  callbacks.push(cb);
  return () => { /* no-op */ };
}
```

Every call creates a fresh local `callbacks` array, pushes the callback into it, and returns a no-op unsubscribe. The callback is retained by the array (which is retained by the closure of the return function), but nothing ever iterates the array so the callback can never fire. The unsubscribe doesn't remove it. Each subscription holds a reference to the callback and whatever it closes over indefinitely. The fix is to either not retain the callback at all (just return a no-op immediately) or maintain a shared instance-level array that mock inbound messages can actually be dispatched through.

## Silent fallback to mock when initialize() is skipped

If `initialize()` is never called and the provider was constructed without an injected client, `this.client` remains null even in live mode. The `sendSMS` method checks `this.liveMode && this.client && this.config` ã with `this.client` null, the condition is false and the method silently falls through to the mock path, returning a `SM_mock_*` SID. A production caller that forgets to await `initialize()` would get mock behavior with no warning or error. This fail-open path should at minimum log a warning, or throw if `liveMode` is true but `client` is null.

## Test coverage gaps

Not tested: what happens when the Twilio client throws during `sendSMS` in live mode (the agent catches it, but the provider doesn't wrap the call ã an uncaught network error would propagate as an untyped exception). Not tested: concurrent access to the message store (relevant if webhooks and user actions race). Not tested: the silent fallback to mock mode when `initialize()` is skipped in live mode.
