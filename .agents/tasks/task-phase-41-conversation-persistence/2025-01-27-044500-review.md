# Conversation persistence layer for quant-live

A new persistence module adds session CRUD, artifact linking, full-text search, LLM-context resume, and an audit event log to the `quant-live` package. The implementation is entirely in-memory (Map-based) with no durable backend, which means the `SessionStore` interface will carry all the weight when a real backend is plugged in. The module ships 5 classes, 1 interface, and 6 new types, all properly exported.

**Watch for:** The `SessionSearch.search` method loads all user sessions into memory before filtering (confirmed). `SessionAudit` is a standalone in-memory log disconnected from the `SessionStore`, so audit events and session data can drift apart (confirmed). `buildConversationHistory` maps `speaker` directly to `role` but the `TranscriptSegment.speaker` type is `'user' | 'assistant'` so it happens to match today â€” any future speaker value (e.g. `'system'`) would silently produce an invalid role (possible).

## High-level view

The `SessionStore` interface is minimal â€” five async methods covering CRUD plus a `list` with pagination and state-filtering. `InMemorySessionStore` adds a `search` method not on the interface, which means consumers who depend on search cannot program to the interface and will be locked to the in-memory implementation or must cast.

`SessionSearch` provides ranked text search with date/artifact-type filters. It delegates to `store.list()` to fetch all sessions for a user, then iterates them in-process. This is O(sessions \* segments) with no early-exit beyond `continue` statements. The interface doesn't expose a way for a real backend to push these filters down, so a future Postgres/Redis store would either re-implement `SessionSearch` or accept the same full-scan pattern.

`SessionResume` converts stored transcripts into LLM-ready conversation history by slicing the last N segments and mapping speaker to role. There's no token-budget awareness â€” 50 long segments could blow a context window while 50 short ones waste budget.

`SessionAudit` is a pure append-only event log that lives outside the store. This decouples audit from CRUD lifecycle but means there's no automatic "session_started" event when `store.create()` is called. Audit discipline relies entirely on callers remembering to record events.

`ArtifactLinker` is a thin read-modify-write wrapper around `store.update`. It has no concurrency guard, so two concurrent `link` calls on the same session could drop one artifact.

1. **`search` not on `SessionStore` interface** â€” The `InMemorySessionStore.search()` method is absent from the `SessionStore` interface. Add it to the interface or extract a separate `SearchableSessionStore` interface so consumers can program to an abstraction.

4. **Audit log disconnected from store lifecycle** â€” `SessionAudit` must be manually invoked by callers. Nothing in `InMemorySessionStore.create/update/delete` emits audit events. Consider event hooks on the store or document the integration contract.
5. **No token-budget awareness in `SessionResume`** â€” `maxSegments` caps count but not total text size. A future improvement could accept a `maxTokens` budget and trim segments by estimated token count.

## `search` living outside the interface contract

`InMemorySessionStore` implements a `search(userId, query)` method that performs case-insensitive substring matching across all transcript segments. This method does not appear on the `SessionStore` interface:

```typescript
// session-store.ts â€” interface has 5 methods, none named search
export interface SessionStore {
  create(...): Promise<SessionStoreEntry>;
  get(...): Promise<SessionStoreEntry | undefined>;
  list(...): Promise<{ entries: SessionStoreEntry[]; total: number }>;
  update(...): Promise<SessionStoreEntry>;
  delete(...): Promise<void>;
}
```

The tests in `session-store.test.ts` call `store.search('user-1', 'keyword')` directly on the concrete class. Any code that holds a `SessionStore` reference cannot call `search` without a cast. Either `search` belongs on the interface, or `InMemorySessionStore.search` is dead weight that duplicates `SessionSearch` logic and should be removed.

## Full-scan semantics in SessionSearch

```typescript
const { entries } = await this.store.list(userId);
```

This line in `SessionSearch.search` fetches every session for a user with no pagination. The `SessionStore` interface's `list` method supports `limit`/`offset`, but `SessionSearch` intentionally bypasses pagination by omitting options. A real backend implementing `SessionStore.list` would either need to:

- Return unbounded results when no limit is specified (dangerous at scale), or
- Provide a separate query method with server-side filtering.

## Race condition in ArtifactLinker.link

The `link` method performs a non-atomic read-modify-write:

```typescript
const session = await this.store.get(sessionId); // read
// ... concurrent call could read same state here ...
await this.store.update(sessionId, { artifacts: [...session.artifacts, full] }); // write
```

If two `link` calls execute concurrently for the same session, both read the same artifacts array, both append their item, and the second `update` overwrites the first's append. The `await` on `this.store.get` yields the event loop, opening a window for interleaving even in single-threaded Node.js.

## SessionAudit as a detached log

`SessionAudit` maintains its own `events` array and `idCounter`, completely independent of `SessionStore`:

- Creating a session via `store.create()` does not produce a `session_started` audit entry unless the caller explicitly calls `audit.record('session_started', id)`.
- Deleting a session removes it from the store but audit events referencing that session persist (correct for audit integrity, but means no automatic lifecycle tracking).
- There's no shared clock or ordering guarantee between store operations and audit records.

## TranscriptSegment.speaker to LLM role mapping

`SessionResume.buildConversationHistory` maps `seg.speaker` directly to a `role` field typed as `'user' | 'assistant'`:

```typescript
return session.transcript.slice(-this.maxSegments).map((seg) => ({
  role: seg.speaker,
  content: seg.text,
}));
```

If `TranscriptSegment.speaker` ever gains a `'system'` variant, this mapping would silently produce invalid LLM messages. The return type annotation locks to `'user' | 'assistant'`, so a type error would surface only if someone updates both types simultaneously.

## Test coverage gaps

Not tested: concurrent artifact linking (the race condition), behavior when `store.update` is called with an `id` field in the patch (the implementation spreads then overwrites `id` â€” `{ ...existing, ...patch, id }` â€” which silently discards a mismatched `id` in the patch), and `SessionAudit` query filtering edge cases (e.g., `since` equal to a timestamp, empty query object).
