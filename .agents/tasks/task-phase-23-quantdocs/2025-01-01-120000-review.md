# QuantDocs collaborative document editor backend

A new `apps/quantdocs/` workspace adds a full backend for collaborative document editing: document CRUD with version history, Yjs-based real-time collaboration, presence/cursor tracking, comments and suggestions, five AI-powered writing tools, multi-format export, and templates. The implementation follows existing monorepo conventions (Fastify plugin routes, class-based services, zod validation, `@quant/ai` integration). The workspace integrates cleanly into the pnpm monorepo with proper dependency declarations and TypeScript strict-mode compilation.

**Watch for:** AI route handlers return unvalidated `JSON.parse` output directly to clients (confirmed) — the services validate with zod but the route layer bypasses them. Header injection via unsanitized `doc.title` in `Content-Disposition` (confirmed). Yjs server holds all document state in-process memory with no eviction or persistence boundary (confirmed). Comment listing endpoint skips auth entirely (confirmed).

## High-level view

The document CRUD layer (DocService) implements soft-delete, version snapshots on every update, and single-owner authorization.

The real-time collaboration stack is split into two in-memory services: YjsServer manages CRDT document state and client connection tracking, while PresenceService tracks cursor positions. Both are purely in-memory with no persistence, eviction, or cross-instance synchronization — single-process placeholders that will leak memory under sustained use.

The AI integration exists at two layers that don't connect. Four service classes parse and validate AI responses with zod schemas. The route file (`ai.ts`) reimplements the same AI calls inline, returning raw `JSON.parse` output without schema validation. The services are never instantiated by the routes.

Export produces placeholder binary formats — PDF and DOCX outputs won't open in any reader. The text formats (Markdown, HTML, LaTeX) are functional.

The test suite (72 tests across 10 files) covers all services with proper mock isolation. Route-level integration tests are absent.

## AI service/route disconnect

The service layer (`ai-write.service.ts`, `ai-grammar.service.ts`, `ai-translate.service.ts`, `ai-diagram.service.ts`) follows a disciplined pattern: call `ai.infer()`, parse JSON, validate with a zod schema, throw a typed error on failure. The route layer (`routes/ai.ts`) reimplements the same AI calls but skips validation:

```typescript
// routes/ai.ts — every handler ends with this pattern:
let parsed: unknown;
try {
  parsed = JSON.parse(response.content);
} catch {
  throw createAppError('Failed to parse AI response', 500, 'AI_PARSE_ERROR');
}
return reply.send({ success: true, data: parsed });
```

The `parsed` value is whatever the LLM returned. If it returns extra fields, wrong types, or a completely different shape, that flows directly to the client. The service classes that validate outputs (`WriteFromOutlineResultSchema`, `GrammarCheckResultSchema`, etc.) are imported in tests but never instantiated in route handlers.

Clients cannot rely on the response shape being stable, and malformed AI output surfaces as arbitrary JSON rather than a controlled error.

## Header injection in export filenames

In `routes/docs.ts`, the export endpoint interpolates `doc.title` directly into the Content-Disposition header:

```typescript
.header('content-disposition', `attachment; filename="${doc.title}.pdf"`)
```

Document titles are user-controlled strings. A title containing `"` breaks the header syntax. A title containing `\r\n` enables HTTP header injection (response splitting). The fix is to either strip non-alphanumeric characters or use RFC 5987 `filename*` encoding.

## Unauthenticated comment listing

`GET /docs/:id/comments` is the only endpoint that skips the auth check:

```typescript
fastify.get<{ Params: { id: string } }>('/:id/comments', async (request, reply) => {
  const prisma = (fastify as unknown as { prisma: unknown }).prisma;
  const service = new CommentService(prisma as never);
  const comments = await service.getComments(request.params.id);
  return reply.send({ success: true, data: comments });
});
```

Every other endpoint in the file extracts `userId` and throws 401 if absent. This endpoint exposes comment content (which may contain private discussion) to anyone who knows a document ID.

## Yjs in-memory lifecycle

`YjsServer` stores every document that's ever been opened as a `Y.Doc` in a `Map<string, Y.Doc>` with no eviction path. The `removeConnection` method cleans up the client set but not the `Y.Doc` itself:

- No eviction when all clients disconnect
- No persistence layer to snapshot state before eviction
- No memory limit or LRU strategy

The `docs` map only grows. For a collaborative editor handling many documents over time, this will eventually OOM the process. The structural fix is to evict (and optionally persist) Y.Doc instances when their connection count drops to zero.

## Presence cursor staleness

`PresenceService` stores cursor positions with a `lastUpdated` timestamp but never reads that timestamp for eviction. If a client disconnects without triggering `removeCursor` (network partition, browser crash, server restart), the stale cursor entry persists forever. Other users would see a phantom cursor for a disconnected collaborator. A periodic sweep comparing `lastUpdated` against a TTL threshold would address this.

## Export format fidelity

`exportToPdf` generates a text string that starts with `%PDF-1.4` but isn't a valid PDF — the xref table offsets are fabricated and content stream lengths are incorrect. `exportToDocx` produces raw WordprocessingML XML without the ZIP container that DOCX requires. Neither will open in any reader application. If these are stubs awaiting a real library (pdfkit, docx), they should be marked as such. If they're meant to be functional, they need a real implementation.

## Prisma availability at runtime

Every route handler in `docs.ts` reaches into `fastify` for a `prisma` property that nothing in this PR decorates:

```typescript
const prisma = (fastify as unknown as { prisma: unknown }).prisma;
const service = new DocService(prisma as never);
```

At runtime, `prisma` will be `undefined` unless an upstream plugin decorates it. The `as never` cast silences TypeScript but hides the fact that DocService would receive `undefined` as its Prisma client, causing runtime errors on the first database call.
