# Next.js API proxy layer and AppShell for QuantSync + QuantAds

This change wires up 54 Next.js App Router API route handlers across two apps (37 for QuantSync, 17 for QuantAds) that proxy frontend requests to their respective backend services. It switches both apps' frontend API clients from direct `localhost` backend calls to relative `/api/*` paths, adds AppShell with Sidebar navigation, creates sub-pages (6 for QuantSync, 2 for QuantAds), removes conflicting legacy `pages/` files from QuantAds, and adds `'use client'` directives to 30 shared-ui components/hooks that were missing them.

Watch for: All proxy routes have **no error handling** around the backend `fetch` call â€” if the backend is unreachable or returns non-JSON, every route will throw an unhandled exception (confirmed). Several API client methods reference routes that don't exist in this PR and will 404 at runtime (confirmed). The auth proxy sends an empty `Authorization` header when no token is present rather than omitting it (confirmed).

## High-level view

The proxy routes follow an identical mechanical pattern: read the request, forward it to `BACKEND_URL` with the Authorization header, parse the JSON response, return it. When the backend is unreachable or returns non-JSON â€” network timeout, 502 from a load balancer, HTML error pages â€” every route will crash with an unhandled rejection because `res.json()` is called unconditionally without try/catch or content-type check. At 54 routes this is a systemic gap, not an edge case.

The API client refactoring switches `API_BASE` from `process.env.NEXT_PUBLIC_API_URL || 'http://localhost:PORT'` to `'/api'`. Eight client methods reference paths without corresponding proxy routes (`/feed/engagement`, `/posts/repost`, `/posts/quote`, `/ai/*`, `/billing/balance`, `/placements/*`) and will 404 at runtime.

The shared-ui changes are a bulk `'use client'` addition plus Prettier reformatting â€” no logic changes, required to unblock the Next.js App Router build.

## Proxy routes: no error boundary around backend fetch

Every route handler follows this exact pattern:

```typescript
const res = await fetch(`${BACKEND_URL}/path`, { headers: { Authorization: ... } });
const data = await res.json();
return NextResponse.json(data, { status: res.status });
```

Two failure modes are unhandled:

1. **Network failure** (backend down, DNS failure, connection refused): `fetch` throws a `TypeError`. No try/catch exists, so Next.js returns a generic 500 with no structured body. The frontend API client expects an `ApiResponse<T>` shape â€” it'll get an HTML error page or nothing.

2. **Non-JSON response** (backend returns HTML error page, 502 from a reverse proxy): `res.json()` throws a `SyntaxError`. Same outcome.

Confirmed across all 54 route files. A single shared utility (`proxyGet`, `proxyPost`, etc.) with try/catch would eliminate the repetition and the gap simultaneously.

- `getPlacementSpecs` â†’ `GET /placements/specs` â€” no route

## Auth token passthrough sends empty header

The pattern `request.headers.get('Authorization') || ''` sends an `Authorization` header with an empty string value when no token is present, rather than omitting it. Backend frameworks that parse the Authorization header (Express + passport, Fastify auth plugins) may treat an empty header as a malformed token and return 401 instead of falling through to anonymous flows.

The fix is to conditionally include the header:

```typescript
const headers: Record<string, string> = {};
const auth = request.headers.get('Authorization');
if (auth) headers['Authorization'] = auth;
```

This affects all 54 routes uniformly.

## No test coverage for route handlers

None of the 54 new route handlers have test files. Given the identical pattern, a single parameterized test that mocks `fetch` and verifies the proxy behavior (happy path + backend-down + non-JSON response) would cover the entire surface and catch regressions if the pattern is later refactored into a shared utility.
