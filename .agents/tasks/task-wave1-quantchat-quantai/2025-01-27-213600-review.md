# Real-time WebSocket wiring + Snapchat screens for QuantChat, Model Selector + Agentic Layer for QuantAI

Two apps gain significant new functionality. QuantChat adds a `RealtimeProvider` wrapping `@quant/realtime`'s `WebSocketClient`, a `useRealtimeChat` hook that dispatches on incoming WS events, and three Snapchat-style screens (Stories with full-screen viewer, Camera with filter carousel, Map with friend locations). QuantAI adds a multi-provider model selector persisted to localStorage, inline tool-call cards with collapsible JSON viewers, an agentic message wrapper, a voice toggle button, and an "Ask Quant" cross-app orchestration placeholder page.

**Watch for:** The WebSocket connection initializes with an empty-string token on first render and never re-establishes when a token becomes available (confirmed). The realtime chat hook grows its `incomingMessages` array unboundedly with no mechanism to merge into fetched history or cap size (confirmed). The QuantAI page renders agentic messages in a separate panel below the chat rather than inline within the message stream, creating a disconnected UX for tool-call-bearing responses (confirmed). Tests for `useRealtimeChat` do not exercise the actual hook; they manually duplicate the branching logic in plain imperative code (confirmed).

## High-level view

The WebSocket integration follows a clean provider/context/hook layering, but the provider creates the connection once with whatever credentials exist at mount time and never reacts to auth state changes. The `useRealtimeChat` hook accumulates state without bounds and has no typing-stop timeout.

The Snapchat-style screens are well-composed with emerald brand theming, proper loading/error/empty state guards, and correct BottomNav routing. The StoryViewer's auto-advance timer risks stale closures if parent callbacks aren't reference-stable.

QuantAI's model selector and tool-call system are cohesive, but the main page places agentic tool-call output in a separate footer panel rather than composing it inline per-message inside the AIChat stream ŌĆö likely because the shared `<AIChat>` component doesn't support custom render slots.

Tests lean on `renderToStaticMarkup` (no jsdom), which validates output markup but cannot test interactivity. The realtime-chat tests duplicate hook internals in plain imperative code rather than exercising the hook itself. The `useAIChat` hook (SSE parsing, abort handling, tool-call accumulation) has zero test coverage.

1. **Empty-token WebSocket connection** ŌĆö `RealtimeProvider` calls `getAuthToken() || ''` once at mount. ## WebSocket provider lifecycle and auth gap

`RealtimeProvider` runs a single `useEffect` with an empty dependency array, meaning the `WebSocketClient` is constructed exactly once. The token comes from `getAuthToken() || ''` ŌĆö if the user lands on the app before auth completes (common with SSR hydration or delayed OAuth flows), the connection authenticates with an empty string. The `autoReconnect: true` flag handles network drops, but it reconnects with the same stale config object; there's no path to inject a fresh token after the initial mount.

```tsx
useEffect(() => {
  const token = getAuthToken() || '';  // stale after this point
  const client = new WebSocketClient({ url, token, ... });
  client.connect();
  return () => { client.disconnect(); };
}, []);  // never re-runs
```

The `publish` callback silently drops messages if the client exists but isn't connected ŌĆö it guards against `clientRef.current === null` but not against `connectionState !== 'connected'`.

## Unbounded message accumulation in useRealtimeChat

Every `message:new` event appends to `incomingMessages` via `setIncomingMessages(prev => [...prev, msg])`. The `incomingMessages` array never shrinks. In a long-running conversation or group chat, this grows without bound.

The `typingUsers` array has no timeout. If a `typing:stop` event is lost (dropped connection, race condition), the UI shows a perpetually-typing user. A 5-second client-side expiry per user entry is standard practice.

## Agentic message placement in QuantAI

The main page renders agentic messages in a `max-h-60` overflow panel _below_ the chat:

```tsx
{messages.some(m => m.toolCalls?.length > 0) && (
  <div className="... max-h-60">
    {messages.filter(m => m.role === 'assistant' && m.toolCalls?.length > 0)
      .map(m => <AgenticMessage ... />)}
  </div>
)}
```

A user sees tool-call cards divorced from their position in the conversation. The likely reason is that `<AIChat>` from `@quant/shared-ui` expects a simple `{role, content}` message format and doesn't support custom render slots ŌĆö an architectural limitation worth addressing to achieve the inline goal.

## StoryViewer auto-advance closure risk

The auto-advance timer depends on callback props `onNext` and `onClose`:

```tsx
useEffect(() => {
  const timer = setTimeout(() => {
    if (currentIndex < stories.length - 1) onNext();
    else onClose();
  }, currentStory.duration * 1000);
  return () => clearTimeout(timer);
}, [currentIndex, currentStory, autoAdvance, isOpen, onNext, onClose, stories.length]);
```

The dependency array includes `onNext` and `onClose`, so the timer restarts whenever the parent provides a new reference. If `useStories()` doesn't return stable references (via `useCallback`), the timer resets every render, preventing auto-advance from ever firing. This is a latent bug that depends on the stability of the calling code.

## Duplicated tailwind color scales

The tailwind config defines literal `emerald`, `indigo`, and `amber` color scales (50-950) that duplicate Tailwind's built-in palette. These add ~50 lines of static config. If the intent is to ensure availability regardless of Tailwind's `content` purging, the simpler fix is to safelist these colors or confirm the default palette is included.

## Test coverage gaps

**Not tested:** No interactivity tests (clicking ModelSelector dropdown, toggling VoiceToggle, expanding ToolCallCard). The `useAIChat` hook ŌĆö the most complex piece of logic in this diff with SSE parsing, abort handling, and tool-call accumulation ŌĆö has zero test coverage.

**QuantChat tests** manually replicate the branching logic of `useRealtimeChat` rather than testing the hook. If the hook's event-dispatch logic is refactored (different field access paths, renamed payload keys), these tests pass regardless. The StoryViewer navigation tests verify simple increment/decrement arithmetic with no connection to the actual component. These tests satisfy a "tests exist" gate but provide minimal regression safety.

## Accessibility gaps

The `ModelSelector` dropdown has no `role="listbox"` or `role="menu"`, no `aria-activedescendant`, and no keyboard navigation (Escape, arrow keys). The ToolCallCard expand/collapse button has no `aria-expanded`. The StoryViewer's tap-to-navigate region has no keyboard equivalent ŌĆö the full-screen overlay is only navigable via mouse click or touch.

## BottomNav routing duplication

Every new page (Stories, Camera, Map, main chat list) duplicates the same `navItems` array and `routes` map ŌĆö 15 lines repeated four times. If a route changes or a nav item is added, all four files need updating. Extract to a shared constant.
