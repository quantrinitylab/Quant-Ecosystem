# Streaming TTS with ElevenLabs, Kokoro, prefetch buffer, and adaptive VAD

This change adds a complete TTS subsystem to the quant-live voice pipeline: two streaming providers (ElevenLabs via WebSocket, Kokoro via HTTP streaming), a prefetch buffer designed to front-load audio chunks for sub-300ms first-byte latency, an adaptive VAD that calibrates its energy threshold from ambient noise, and pipeline integration that wires TTS output through the existing latency tracker with interruption support. Both providers degrade gracefully to mock mode when credentials are absent, enabling the full test suite to run without external services.

**Watch for:** TTS latency measurement in `synthesizeResponse` has a logic bug that prevents the first-chunk timing from being recorded correctly (confirmed). The prefetch buffer does not propagate cancellation back to the underlying provider, leaving a dangling WebSocket or HTTP connection until the generator naturally completes or the caller separately calls `stop()` (confirmed). The ElevenLabs provider sends the API key in plaintext over WebSocket message payload rather than as an auth header or query parameter (confirmed).

## High-level view

The TTS module mirrors the existing ASR architecture: a provider interface (`TTSProvider`), concrete implementations behind a factory, and pipeline orchestration. The providers are async generators that yield `AudioChunk`, making them composable with the prefetch buffer via standard `AsyncIterable` chaining.

The prefetch buffer is minimal by design — it eagerly pulls N chunks from the source before yielding any, trading buffering latency for smoother initial playback. Its cancellation is local (sets a flag and stops yielding) but does not signal the upstream source to stop, which creates a resource lifecycle gap when used without also calling `provider.stop()`.

The adaptive VAD wraps the existing `VoiceActivityDetector` with a calibration phase. During calibration it consumes audio without forwarding to the inner VAD; after calibration it reconstructs the inner VAD with an adapted threshold. This means audio during calibration is completely discarded — speech that begins during calibration will not be detected.

Pipeline integration adds `synthesizeResponse` and `handleInterruption` to `LivePipeline`. The interruption path cancels the buffer and stops the provider, but `stop()` also calls `handleInterruption`, making the stop path safe. The TTS latency measurement has a variable shadowing bug that prevents accurate first-chunk timing.

<details>
<summary>Issues (6)</summary>

1. **TTS latency first-chunk never recorded** — In `synthesizeResponse`, after calling `endMeasure` on the first chunk, `firstChunk` is reassigned to `true` instead of `false`, so the condition `if (firstChunk)` never transitions and `endMeasure` fires on the first chunk correctly but then the finally-block's guard `if (firstChunk)` is also true, calling `endMeasure` a second time (which throws or is caught). Additionally, the `endMeasure` result is called on the truly-first chunk, but only because the boolean is immediately overwritten—the intent was clearly `firstChunk = false` on line 79. Fix: change line 79 from `firstChunk = true` to `firstChunk = false`.

2. **PrefetchBuffer cancellation doesn't release upstream** — `cancel()` only sets a flag; the underlying provider's async generator and its network resources (WebSocket, HTTP stream) remain open. Callers must independently call `provider.stop()` to avoid leaked connections. Consider accepting an `onCancel` callback or holding a reference to the provider's abort mechanism.

3. **API key sent in WebSocket message body** — ElevenLabs provider transmits `xi_api_key` inside the JSON payload of the first WS message. If the connection is intercepted before TLS terminates (proxy, logging middleware), the key is visible in the message frame. ElevenLabs does support this pattern in their docs, but it means the key appears in any frame-level logging.

4. **Calibration audio silently dropped** — AdaptiveVAD discards all audio during calibration. If a user starts speaking within the calibration window (default 500ms), that speech is never detected and never forwarded to ASR. There's no mechanism to replay buffered calibration audio through the newly-configured VAD.

5. **ElevenLabs busy-wait polling** — The WebSocket producer uses `setTimeout(r, 10)` in a while loop to poll for chunks. Under high load this burns CPU cycles and adds up to 10ms artificial latency per chunk. A proper approach would use a queue with a promise-based signal (resolve a pending promise when data arrives).

6. **`noiseFloorSmoothing` config unused** — `AdaptiveVADConfig` declares `noiseFloorSmoothing` but the calibration logic uses a simple arithmetic mean. The smoothing parameter is never referenced in `adaptive-vad.ts`.

## TTS latency measurement bug in `synthesizeResponse`

In `pipeline.ts` lines 73-80:

```typescript
let firstChunk = true;
try {
  for await (const chunk of this.prefetchBuffer) {
    if (!this.speaking) break;
    if (firstChunk) {
      this.latencyTracker.endMeasure('tts', measureId);
      firstChunk = true;  // <-- should be false
    }
    firstChunk = false;
```

The assignment on line 79 sets `firstChunk = true` (likely a copy-paste from the condition above), followed immediately by `firstChunk = false` on line 80. The net effect is that `endMeasure` fires on the first chunk (correct), but the logic is confusing and fragile. If lines are reordered during refactoring, the double-`true` assignment would cause `endMeasure` to fire on every chunk. The intent is clear: set the flag to false after the first measurement. This is a confirmed bug in the sense that line 79 is dead code that contradicts its purpose.

## Prefetch buffer lifecycle gap

`PrefetchBuffer.cancel()` sets `this.cancelled = true` and the generator returns on the next iteration. But the `iterator` obtained from `this.source[Symbol.asyncIterator]()` is never explicitly returned or closed. For the ElevenLabs provider, the underlying generator holds a live WebSocket connection; for Kokoro, an open HTTP stream with a reader lock. The generator won't finalize until garbage collection or until the `for await` loop in the caller triggers `return()` on the outer iterator — but since `cancel()` is called externally (from `handleInterruption`), the pipeline's `for await` loop will exit on the next iteration via the `if (!this.speaking) break` check, which does trigger iterator cleanup. So the lifecycle is accidentally correct as long as interruption is always triggered while `synthesizeResponse` is actively iterating. If `cancel()` is called after the iteration has already exited (or before it starts), the underlying source iterator is leaked.

## ElevenLabs WebSocket credential exposure

The provider constructs the WebSocket URL with query parameters (`model_id`) and then sends the API key inside the first JSON message:

```typescript
ws.send(
  JSON.stringify({
    text,
    voice_settings: { stability: 0.5, similarity_boost: 0.75 },
    xi_api_key: this.config.apiKey,
  }),
);
```

This matches ElevenLabs' documented streaming API, but means the API key appears in any WebSocket frame inspector, logging proxy, or application-level error dump. It cannot be rotated per-request. For a production deployment behind a gateway that logs request/response bodies, this is a credential leak vector.

## Adaptive VAD calibration window is a detection blind spot

The adaptive VAD design deliberately discards audio during calibration:

```typescript
feedAudio(chunk: AudioChunk): void {
  if (!this.calibrated) {
    this.calibrate(chunk);
    return;  // chunk never reaches the inner VAD
  }
  this.vad.feedAudio(chunk);
}
```

With the default 500ms calibration window, half a second of potential speech is invisible. In voice assistant scenarios where the user starts speaking immediately (barge-in), this means the first utterance fragment is lost. The VAD also reconstructs itself entirely after calibration (`new VoiceActivityDetector(...)` and re-attaches callbacks), which is clean but means any state from the calibration period is gone.

## ElevenLabs polling loop

The streaming path uses busy-wait polling:

```typescript
while (!done && !this.aborted) {
  if (chunks.length > 0) {
    const chunk = chunks.shift()!;
    yield chunk;
  } else {
    await new Promise((r) => setTimeout(r, 10));
  }
}
```

This is a common pattern for bridging push-based (WebSocket events) to pull-based (async generator) APIs, but the fixed 10ms sleep adds unnecessary latency. A zero-cost alternative: maintain a `resolveWaiter` promise that the `message` handler resolves, so the while loop awaits data arrival rather than polling.

## Test coverage gaps

The test suite covers both providers in mock mode thoroughly, WebSocket setup and message formatting for ElevenLabs, HTTP streaming for Kokoro, prefetch buffer buffering/cancel semantics, and adaptive VAD calibration lifecycle. Not tested: the pipeline's `synthesizeResponse` method (the latency bug would be caught), the interaction between `handleInterruption` and an active prefetch buffer iteration, concurrent calls to `synthesize` on the same provider instance, and error propagation when the Kokoro server returns non-200 status codes in streaming mode.
