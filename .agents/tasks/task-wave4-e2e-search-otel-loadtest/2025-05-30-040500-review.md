# Wave 4: E2E tests, Meilisearch engine, OpenTelemetry SDK, k6 load testing, admin observability page

Wave 4 adds five capabilities to the monorepo: Playwright-based E2E tests for admin and critical flows, a Meilisearch/InMemory search engine factory behind a `SearchEngine` interface, real OpenTelemetry tracing+metrics wired through a Fastify plugin that activates only when `OTEL_EXPORTER_OTLP_ENDPOINT` is set, k6 load test scenarios (smoke/load/stress) with SLO definitions, and a static admin observability dashboard page. Docker-compose gains Meilisearch, Jaeger, Prometheus, and Grafana services. The OTel integration is properly opt-in, the search factory is clean, and the observability plugin uses dynamic imports so services that never set the env var pay no cost.

**Watch for:** The k6 auth helper produces tokens with an invalid signature (the "HMAC" is just a base64 of the signing input—**confirmed**), which means load tests will fail against any server that validates JWT signatures. The `MeilisearchEngine.search` silently swallows errors and returns empty results—intentional for resilience, but means search failures are invisible unless someone reads console logs (**confirmed**). The CI e2e job has no build step and no server startup, so it will fail on any runner that doesn't pre-run the app (**confirmed**).

## High-level view

The search engine abstraction is a straightforward factory: `createSearchEngine()` reads `MEILISEARCH_URL` and returns either a `MeilisearchEngine` or an `InMemoryEngine`. The `InMemoryEngine` does naive substring matching rather than using the existing `BM25Ranker` the feature spec called for—results aren't ranked by relevance, and a doc matching one of five terms ranks the same as one matching all five.

The OTel integration is layered as SDK primitives (`initTracing`, `initMetrics`, `createLogger`) in `@quant/observability` and a Fastify plugin in `server-core` that conditionally wires them via dynamic `import()`. A module-level `let sdk` singleton means multiple plugin registrations (e.g., in test isolation) would silently overwrite each other.

The CI e2e job shards across 4 runners but lacks the build and serve steps needed to have an app running at `localhost:3000`. The k6 load testing generates fake JWT tokens whose signature is not HMAC—they will be rejected by any real auth layer.

Prometheus scrape config targets `/metrics` on each service, but no service currently exposes a `/metrics` endpoint (the OTel plugin exports via OTLP push, not Prometheus pull). The admin observability page renders entirely from hardcoded mock data.

## Search engine factory and the InMemory/Meilisearch split

`MeilisearchEngine` wraps every SDK call in try/catch. For `search`, errors produce an empty result and a `console.error`—a misconfigured Meilisearch URL degrades silently with no structured signal. For mutations (`indexDocuments`, `removeDocuments`, `createIndex`, `deleteIndex`), errors are logged _and_ rethrown—reads degrade, writes fail loudly. This asymmetry is deliberate but the silent read-failure path needs a metric or log level above "error in a console.error" to be actionable in production.

The `totalHits` extraction uses a cast through `Record<string, unknown>` because the SDK's `SearchResponse` type changed across versions—`totalHits` was renamed from `nbHits`. The fallback to `estimatedTotalHits` handles older SDK responses but will break silently if neither field exists in a future SDK version.

`InMemoryEngine.indexDocuments` without a prior `createIndex` call silently creates the store but never records a config, so `primaryKey` defaults to `'id'`. Documents without an `id` field will be stored with key `''` and overwrite each other. This path exists because `indexDocuments` lazily initializes storage (`if (!this.storage.has(index))`) but reads `this.configs.get(index)` which returns `undefined`.

## OTel SDK: singleton leak and metrics provider one-shot

`initTracing` stores the SDK in a module-level `let sdk`. Calling it a second time without calling the returned shutdown function overwrites `sdk` and leaks the first instance—there's no guard. In production this is unlikely (single plugin registration), but in tests with per-suite setup, it accumulates leaked providers.

`initMetrics` has the same pattern and additionally calls `metrics.setGlobalMeterProvider(meterProvider)` which is a one-shot operation in the OTel API—calling it again logs a warning but doesn't replace the provider. So the second call creates a new MeterProvider that never actually becomes the global one.

The server-core observability plugin dynamically imports `@quant/observability` only when the endpoint is set. The `onRequest` hook stamps a start time via an unsafe cast (`request as unknown as Record<string, number>`). If any other plugin stamps the same `__otelStartTime` key, they'll collide—unlikely but a `Symbol` key would be safer.

## CI e2e job: will fail without a server

The workflow step runs `pnpm test:e2e -- --project=chromium --shard=${{ matrix.shard }}` with `BASE_URL: http://localhost:3000`. No step builds the app or starts a server. Playwright will navigate to `localhost:3000` and get connection refused.

A working pipeline needs either a Playwright `webServer` config in `playwright.config.ts` that starts the app, or explicit build + start steps in the workflow.

## k6 auth tokens: structurally valid, cryptographically invalid

`loadtest/k6/helpers/auth.js` constructs a JWT with the right structure (header.payload.signature), but the signature is `b64encode(signingInput)` rather than `HMAC-SHA256(signingInput, secret)`. The comment acknowledges k6 lacks native HMAC.

The k6 runtime _does_ provide `k6/crypto` with `hmac('sha256', secret, data, 'hex')` which could produce valid signatures. Alternatively, the README should document that tokens must be pre-generated or auth bypassed for load testing.

## Prometheus scrape config vs OTLP push

`docker/prometheus/prometheus.yml` defines scrape targets for all services at `/metrics`. The OTel integration uses OTLP push (`OTLPMetricExporter` posting to Jaeger's collector). No service exposes a `/metrics` HTTP endpoint, so Prometheus will scrape and get 404s.

For Prometheus to work, either add a `PrometheusExporter` endpoint (pull-based) to the server-core observability plugin, or point Prometheus at an OTLP-to-Prometheus adapter.

## E2E fixture coupling: TEST_AI_SESSIONS reference

`ask.spec.ts` imports `TEST_AI_SESSIONS` from `../../fixtures/test-data`. The visible exports from test-data include `TEST_USERS`, `TEST_CONVERSATIONS`, `TEST_EMAILS`, and several others—but `TEST_AI_SESSIONS` is not in the first 60 lines. If it was added further down the file, this works. If it wasn't, the test will throw `TypeError: Cannot read properties of undefined (reading 'basic')` at runtime. Verify this export exists.

The admin dashboard specs duplicate the login flow inline rather than using the shared `login()` helper from auth-helpers—a minor consistency issue but not a bug.
